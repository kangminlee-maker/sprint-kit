# onto_evolution — sprint-kit 도메인 학습

- 이벤트 소싱 시스템에서 "재개" 기능을 추가할 때, 재개 시점에 외부 입력이 변경되었을 경우의 처리를 반드시 정의해야 함. 이미 기록된 이벤트는 수정 불가능하므로, 변경을 반영하려면 새 이벤트(역전이 등)가 필요 (출처: 10건 피드백 구현 계획 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- 이벤트 소싱 시스템에서 "이벤트 분류 변경"과 "상태 추가"는 독립된 결정. 묶어 제안하면 반대자에게 더 큰 표적을 제공. 분리하여 제안하면 합의 범위를 최대화 (출처: exploration 승격 리뷰, 2026-03-13) (→ 글로벌 승격 완료, 2026-03-15)
- 검증 시점과 데이터 확정 시점을 분리하면 새 검증 계층의 파이프라인 삽입 위치가 자연스럽게 결정됨 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- "N개의 별도 계층"보다 "하나의 시점에서 N가지 관점"으로 재정의하면 상태 추가, 타입 분리, 이벤트 추가 등 복잡도 증가를 모두 회피 가능 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- "개념적 필요성 합의"와 "별도 파일/모듈 구현"은 별개 판단. 5인 개념 합의가 구현 합의로 무의식 확장되는 위험 (출처: 산출물 온톨로지 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- CST→IMPL→CHG→VAL 추적 체인에서 IMPL/CHG/VAL은 Compile 시점에 일괄 생성되고 재시도 시 전체 교체됨. 이 생명주기는 관계형 모델의 "개별 레코드 CRUD"와 맞지 않으며, foreign key로 관계를 강제하면 교체 시 cascading delete가 필요하여 이벤트 소싱의 추가 전용 원칙과 충돌 (출처: Schema-as-Ontology 리뷰, 2026-03-14)
- Sprint Kit의 지식은 "구조적 지식(엔티티/관계) + 행동적 지식(규칙/원리)"으로 분류됨. 구조적 지식은 types.ts에 이미 코드화되어 drift 위험이 낮고, 행동적 지식은 변경 빈도가 낮아 drift 위험이 낮음. 별도 관리 인프라(knowledge/ 분리)는 drift가 실제로 관찰된 시점에 대응하는 것이 확장성을 유지 (출처: Schema-as-Ontology 리뷰 Round 2, 2026-03-14)
- [사실] Sprint Kit의 도메인 개념(Constraint, Source, Scope 등 14개+)과 파일시스템 구조 사이에는 N:M 관계가 존재함. 파일시스템의 1D 트리 구조는 이 N:M 관계를 표현할 수 없으므로, 도메인별 디렉토리 재구성은 어느 한 축에서 반드시 정보 손실을 발생시킴. 역방향 인덱스(문서)가 트리를 유지하면서 N:M 관계를 표현하는 적합한 접근 (출처: 파일시스템 구조 리뷰, 2026-03-15)
