# onto_structure — sprint-kit 도메인 학습

- "정의 권한"과 "명세 권한" 이원 구조 도입 시, 변경 방향 규칙(정의→명세→코드)과 기계적 대응 관계 기록이 없으면 관리 문서만 증가 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09)
- Constraint를 Function 내부에 내장하면 N:M 관계 기록 위치가 모호. Local(1 Function)/Global(2+ Functions) 분류 하이브리드로 해소 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09)
- "상세도의 종류"(의미/구조/구현)와 "상세도의 배치 방식"(Layer 1/2/3)은 독립된 축. 혼동하면 불필요한 충돌 발생 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09)
- "정보 구조와 렌더링 분리" 원칙이 합의되었을 때, 기존 타입 시스템(ContentType)이 이미 IR 역할을 수행하는지 먼저 확인. 문자적 해석과 실질적 달성을 구분하지 않으면 과설계 (출처: 산출물 온톨로지 리뷰, 2026-03-09)
- 중간 추상 계층은 그 계층의 인스턴스를 직접 소비하는 소비자가 2개 이상 존재할 때만 정당화. 소비자 1개이면 간접 비용만 발생 (출처: 산출물 온톨로지 리뷰, 2026-03-09)
- 이벤트 소싱 시스템에서 식별자와 저장소 경로를 동일하게 사용하면 식별자 불변성 요구와 경로 변경 요구가 충돌. 분리 비용보다 마이그레이션 설계가 더 중요한 결정 요소 (출처: 10건 피드백 구현 계획 리뷰, 2026-03-09)
- 검증 계획을 AI에게 위임할 때, "무엇을 검증하라"(pass_criteria)와 "어떻게 검증했는지"(evidence_type)는 별도 필드로 분리해야 감사 추적 가능 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12)
- edge case가 검증 계획에 열거되어도 결과를 기록하는 구조화된 경로가 없으면 "문서화된 edge case"와 "실제 검증된 edge case"가 분리됨 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12)
- 검증 계층을 "실행 시점"으로 분리할 때, 기존 시스템에 이미 분산 수행되고 있는 검증과의 경계를 먼저 명확히 해야 함 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12)
- 구조적 검증(코드)과 실행 절차(프로토콜)가 다른 문서에 있을 때, 연결 고리를 프로토콜 문서에 역참조로 명시해야 enforcement가 완성됨 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12)
- 하나의 명령이 "자동 실행 단계"와 "사용자 대화 단계"를 모두 포함할 때, 상태 머신에서 다른 상태로 분리하면 세션 재개 복구가 상태 읽기로 단순화됨 (출처: exploration 승격 리뷰, 2026-03-13)
- "정의 문서"와 "명세 문서" 계층 분리 시, 경계 기준은 "이것을 모르면 의도를 잘못 해석하는가?" (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09)
- event-pipeline.ts Step 5의 "이벤트 파일 재읽기 생략(배열 concat)"과 "reduce() 내부의 증분 순회"는 다른 계층의 최적화. 전자는 이미 구현됨. 후자를 도입하려면 reducer.ts의 18개+ 로컬 변수를 이전 상태에서 복원하는 매핑이 필요하며 누락 시 파생 필드(stale, convergence_blocked 등) 불일치 발생 (출처: 속도 개선 설계 구조적 완전성 리뷰, 2026-03-14)
- computeDirectoryHash의 해시는 source_hashes로 grounding 이벤트에 영구 기록되므로, 해시 계산 알고리즘(정렬 순서, 결합 방식) 변경은 이벤트 스키마 호환성 파괴와 동일한 영향. 스캔 해시는 이벤트 소싱 시스템의 불변 데이터 (출처: 속도 개선 설계 구조적 완전성 리뷰, 2026-03-14)
- [사실] SourceEntry union type을 switch하는 함수는 현재 4개: sourceKey() (kernel/types.ts), toGroundingSource() (scanners/types.ts), scanSource() (commands/start.ts), toBriefSourceEntry() (commands/start.ts). scanSource()만 exhaustive check 존재 (출처: 속도 개선 #2-#5 작업설계 리뷰, 2026-03-15)
