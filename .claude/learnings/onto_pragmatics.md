- Primary 소비자가 AI 에이전트일 때, 정보 구조(AI)와 렌더링 계층(사람)을 분리 설계해야 함. AI는 용어 일관성과 물리적 근접성에 민감 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)

- AI 에이전트가 실행 주체인 시스템에서는 필드의 "존재 여부"보다 "실행 가능한 분해 수준"이 품질 기준. AI가 행동을 수행해야 할 때는 절차·사전 조건·판정 기준의 분리가 필요 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- "다른 에이전트가 실행"하는 명세를 작성할 때, 명세는 실행에 필요한 모든 정보를 자체 포함(self-contained)해야 함 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- "검증 가능 시점"과 "검증 실행 시점"은 별개. shift-left verification이 PO 판단력과 효율성을 모두 높임 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- 멀티 독자 단일 문서에서 문서 상단 독자별 읽기 경로 명시 → 각 독자가 자신의 정보 층위 즉시 선택 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- 선택지 설명은 (1) 언제 선택하는가(trigger condition)와 (2) 선택하면 무슨 일이 일어나는가(consequence) 두 정보를 모두 포함해야 함 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)

- 프론트엔드에서 여러 API 응답을 조합하여 화면 상태를 결정할 때, 조합 규칙을 독립된 "상태 판정 테이블"로 분리 필수 (출처: homeUI Align Packet 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)
- AI primary 소비자 문서에서 "정보를 포함하는 것"과 "행동을 지시하는 것"은 별개. "무엇을 확인하라"와 "어떻게 확인하라"를 동시에 명시해야 함 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- 비동기 협업 시스템에서 산출물 파일이 유일한 맥락 전달 경로인 경우, 렌더러 수준에서 행동 지침을 내재화하는 것이 프로토콜 문서보다 신뢰할 수 있는 전달 수단 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- 새 문서를 추가하는 것보다 기존 산출물의 정보 밀도를 높이는 것이 PO 인지 부담을 줄임. "충돌 없음"이라는 긍정 결과도 명시 기록해야 PO가 구분 가능 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- system-driven 활동과 user-driven 활동이 같은 상태 값에 포함되면, 사용자는 "지금 내가 행동해야 하는가"를 상태 값만으로 판단할 수 없음 (출처: exploration 승격 리뷰, 2026-03-13) (→ 글로벌 승격 완료, 2026-03-15)
- 상태 정보가 두 곳에 분산되어 사용자가 조합해야 현재 위치를 파악할 수 있는 구조는, 세션 재개 시점에서 특히 취약 (출처: exploration 승격 리뷰, 2026-03-13) (→ 글로벌 승격 완료, 2026-03-15)
- sprint-kit의 코드 source of truth는 `kernel/types.ts`이지 `blueprint.md`가 아님. 17개 파일이 types.ts를 import하며, PayloadMap이 이벤트 타입-payload 1:1 매핑을 강제. blueprint.md는 인간 독자용 문서. drift가 발생한다면 blueprint.md ↔ types.ts 사이 (출처: Schema-as-Ontology 재구성 리뷰, 2026-03-14)
- 에이전트 프로토콜 8개 파일이 appendScopeEvent() 50회+, 필드명 47회를 직접 참조. 이 구조에서 "진짜 drift"는 프로토콜 문서(.md)의 API 호출 예시와 실제 TypeScript 시그니처 사이에서 발생. 이 drift 검증은 프로토콜 문서에서 코드 스니펫을 추출하여 PayloadMap과 대조하는 스크립트로 대응 가능 (출처: Schema-as-Ontology 재구성 리뷰 Round 2, 2026-03-14)
- TypeScript discriminated union의 switch 문에서 exhaustive check 없이도 반환 타입 명시 시 호출 사이트에서 컴파일 오류 발생. 그러나 오류 위치가 원인(case 누락)이 아닌 소비 지점에 분산됨. exhaustive check(`const _exhaustive: never = x`) 패턴의 핵심 가치는 "오류 위치를 원인 지점에 집중"시키는 것 (출처: 속도 개선 #4 exhaustive check 검증, 2026-03-15) (→ 글로벌 승격 완료, 2026-03-15)
