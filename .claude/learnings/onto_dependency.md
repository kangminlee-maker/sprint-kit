# onto_dependency — sprint-kit 도메인 학습

- 공유 타입이 특정 모듈에 정의되면 의도하지 않은 의존 방향 생성. 의존 계층 최하위에 배치가 구조적으로 안전 (출처: Scanner 설계 리뷰, 2026-03-09)
- 타입 정의의 모듈 위치는 그 타입의 계약 수준(contract level)을 암묵적으로 선언. 소비자가 시스템 외부 경계(에이전트 프로토콜)에 있으면 커널 계약으로 유지가 의도 보존에 유리 (출처: 산출물 온톨로지 리뷰, 2026-03-09)
- 소스 메타데이터(usage_stage 등 운영 관심사)는 커널 타입이 아닌 config 레이어에 유지해야 함. 커널 타입이 이벤트 payload를 통해 이벤트 스트림에 도달하면 불필요한 필드가 영구 기록됨 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10)
- 타입 위치 이동 논의 시, 의존 그래프 방향을 먼저 확인. 타입을 소비자 모듈로 이동하면 생산자가 소비자에 의존하게 되어 구조적 제약 위반 가능 (출처: 산출물 온톨로지 리뷰, 2026-03-09)
- 외부 서비스(MCP 등) 기반 소스의 변경 감지는 서비스 제공 메타데이터(lastModified 등) 기준이 위양성 방지에 유리 (출처: Scanner 설계 리뷰, 2026-03-09)
- 식별자 형식 변경은 "폴더명 변경"처럼 보이지만, 식별자가 참조 무결성 체계를 관통하면 시스템 전체 규모의 변경이 됨 (출처: 10건 피드백 구현 계획 리뷰, 2026-03-09)
- 의존 관계가 있는 다수 결정 항목 처리 시, 기존 단일 처리 흐름의 호출 순서를 제어하는 규칙 추가가 새 구조 도입보다 통합 비용 낮음. 단, 기존 흐름이 충분히 일반적일 때만 성립 (출처: homeUI Align Packet 리뷰, 2026-03-10)
- 데이터 수집 입력 경로가 3곳 이상이면, 각 경로의 소비 시점과 "진실의 원천" 지정이 필수. 선반영(pre-populate)은 1회성 스냅샷으로 한정하면 동기화 문제 회피 가능 (출처: 10건 피드백 구현 계획 리뷰, 2026-03-09)
- 데이터 파이프라인에서 원문과 메타데이터를 분리 저장할 때, 원문이 필요한 하위 단계가 생기면 별도 재취득 경로를 명시적으로 설계해야 함 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10)
- "A가 명세를 쓰고 B가 실행하는 비동기 파이프라인"에서 타입 계약은 세션 간 책임 연결의 실질적 수단. 프로토콜은 같은 세션에서 읽히지만, 타입 계약은 세션과 무관하게 구조로 강제됨 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12)
- reducer가 envelope 필드(state_after)를 신뢰하는지, 이벤트 내용으로부터 재계산하는지는 이벤트 스키마 변경의 영향도를 결정하는 핵심 설계 선택 (출처: exploration 승격 리뷰, 2026-03-13)
- 동일 시스템에서 이벤트 스트림(구조적 상태)과 파일(의미적 맥락) 두 경로로 데이터를 전달할 때, source of truth와 불일치 시 우선순위를 계약으로 명시해야 함 (출처: exploration 승격 리뷰, 2026-03-13)
- source of truth 이전(migration) 시, 기존 문서의 "정의 권한" 규칙과 새 시스템의 "source of truth" 선언이 공존하면 의존 방향이 이중화됨. source of truth를 이전할 때는 기존 문서의 권한 규칙도 함께 갱신해야 함 (출처: Schema-as-Ontology 리뷰, 2026-03-14)
- 문서 분리(knowledge/ 등) 작업의 실행 시점은 "분리할 내용이 준비되었는가"가 아니라 "참조 대상이 확정되었는가"와 "source of truth 규칙이 갱신되었는가"로 판단해야 함. 참조 대상 부재 상태에서의 분리는 참조 무결성을 검증할 수 없게 만듦 (출처: Schema-as-Ontology 리뷰 Round 2, 2026-03-14)
