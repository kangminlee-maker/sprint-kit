# onto_logic — sprint-kit 도메인 학습

- "지능적 범위 축소 → 빈 결과 → 전체 스캔 fallback"의 3단 구조가 신뢰성 기본 패턴. 최적화 도입 전 fallback 먼저 구현이 안전 (출처: Scanner 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- Function 전제 조건에 constraint를 내장하면 코드 대응이 명확하나, 전제 조건 간 실행 순서 의존성은 별도 명시 필요 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- hard/soft constraint 분류 도입 시, 문서에는 있지만 코드에 강제 장치가 없는 규칙을 체계적으로 발견 가능. blueprint 작성 자체가 문서-코드 일치성 감사 역할 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- discriminated union과 string literal union이 다른 파일에서 동일 대상 표현 시, 한쪽 추가가 다른 쪽 컴파일 에러를 유발하지 않을 수 있음. switch의 `default: never` 패턴이 유일한 안전장치 (출처: Scanner 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- 이벤트 소싱에서 단일 비즈니스 작업이 복수 이벤트를 기록할 때, partial commit 방지를 위해 (1) 복합 이벤트 통합, (2) 선행 실행 후 마지막에 기록, (3) saga 패턴 중 하나를 명시적으로 선택해야 함 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)
- 일괄 처리에서 Phase 단위로 트랜잭션 경계를 두고, 각 Phase 완료 시 정합성 검증을 수행하는 것이 실용적 패턴 (출처: homeUI Align Packet 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)
- hard/soft constraint 경계는 단일 질문 판정 어려움. "무효화 여부"+"국소화 가능 여부" 2단계 구조화로 경계 사례 처리 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- 산출물 생성의 제약은 "체크리스트로 사후 검증"보다 "생성 지침에 사전 포함"이 더 강한 품질 보증. 사후 검증은 위반을 감지하지만 차단하지 못함 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)
- "추적 체인 완전성"(A→B→C→D 연결)과 "조합 정합성"(A1+A2 논리적 무모순)은 별개 검증 차원. 전자만으로 후자를 보장할 수 없음 (출처: compile 후 정책 리뷰 잔존 원인 분석, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- LLM이 생성한 산출물에 "모호함 없음" 속성을 검증하는 것은, "모호함 발견 시 차단"으로 재정의해야 실현 가능. 증명(proof)이 아닌 탐지(detection)로 전환 필수 (출처: compile 후 정책 리뷰 잔존 원인 분석, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- 의미적 정합성 검증이 필요할 때, "의미를 코드로 판단"보다 "구조적 패턴으로 의심 지점을 탐지하고 경고"하는 접근이 비결정적 시스템에서 현실적 (출처: compile 후 정책 리뷰 잔존 원인 분석, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- 자유 텍스트 pass_criteria와 expected_result는 AI 에이전트가 실행 주체일 때 "부분 충족"과 "자의적 해석" 두 가지 장애 모드를 동시에 가짐 (출처: validation-plan 업그레이드 설계 리뷰, 2026-03-12) (→ 글로벌 승격 완료, 2026-03-15)
- 이벤트 소싱 시스템에서 "관찰 이벤트 + gate-guard 규칙"은 상태 머신 매트릭스를 변경하지 않고 새로운 검증 게이트를 추가하는 안전한 패턴 (출처: validation-plan 코드 독립적 검증 리뷰, 2026-03-12)
- 복잡한 하위 상태(sub-state)를 observational 이벤트로 구현하면, "진입 조건 위반"과 "완료 전 탈출" 두 가지 비정상 경로가 잠재. gate-guard Rule이 두 경로를 명시적으로 차단해야 함 (출처: exploration 승격 리뷰, 2026-03-13)
- "이력 의존 조건"(이전 이벤트 존재 여부)은 상태 전이 행렬(MATRIX)로 표현할 수 없다. gate-guard 계층에서 처리하는 것이 역할 분담 원칙과 일관 (출처: exploration 승격 리뷰, 2026-03-13)
- observational 이벤트가 정당한 두 조건: (1) 발생 상태에 무관하게 의미가 동일, (2) 다른 이벤트의 허용/거부 판정에 영향을 주지 않아야 함 (출처: exploration 승격 리뷰, 2026-03-13)
- "참조 무결성"(A가 B를 가리키는가)과 "의미적 정합성"(A 내용이 B 내용과 양립)은 별도 검증 계층. 참조 무결성 통과해도 의미적 정합성 미보장 (출처: compile 후 정책 리뷰 잔존 원인 분석, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- discriminated union에서 실패 분기(success:false)에 state 필드를 타입 수준에서 배제하면, "실패 시 부분 상태를 후속 처리에 전달"하는 시나리오를 컴파일 타임에 차단할 수 있음. 이벤트 소싱의 pipeline result 설계에서 유효한 안전장치 (출처: 속도 개선 설계 리뷰, 2026-03-14) (→ 글로벌 승격 완료, 2026-03-15)
- 기존 함수를 내부 함수로 분리(리팩터링)할 때, "정렬 방식"과 "예외 처리 경로(fallback 값)"가 동일 결과 보장의 숨겨진 조건. 정상 경로의 동치성만 검증하면 예외 경로(unreadable 등)의 차이를 놓침. 리팩터링 동치성 검증은 정상/예외 경로를 분리하여 각각 수행해야 함 (출처: 속도 개선 작업설계 리뷰, 2026-03-14) (→ 글로벌 승격 완료, 2026-03-15)
- 작업설계에서 동일 함수명이 복수 파일에 존재할 때, 변경 대상 열거의 검색 범위를 특정 파일로 한정하면 누락이 구조적으로 발생. 전수 조사(grep) 결과를 명시해야 범위 불일치를 방지 (출처: 속도 개선 작업설계 리뷰, 2026-03-14) (→ 글로벌 승격 완료, 2026-03-15)
- 위 원칙의 확장: 동일 함수명이 아닌 **동일 패턴**(같은 discriminated union의 switch without default)으로 검색 범위를 설정해야 누락을 방지. 함수명 기반 전수 조사보다 패턴 기반 전수 조사가 더 넓은 범위를 포착. 예: SourceEntry switch 함수 4개 중 toBriefSourceEntry() 누락 사례 (출처: 속도 개선 #2-#5 작업설계 리뷰, 2026-03-15)
- [사실] sprint-kit의 kernel/types.ts는 21개 파일에서 29회 import됨. ScopeState가 ConstraintPool을 포함하므로 도메인별 디렉토리 분리(scope/ + constraint/) 시 순환 의존이 구조적으로 발생함. 현재의 기술 역할별 분류(kernel/commands/scanners/...)가 단방향 DAG를 유지하는 구조적 기반 (출처: 파일시스템 구조 리뷰, 2026-03-15)
- [사실] sprint-kit의 프로토콜 문서(docs/agent-protocol/)의 코드 조각은 kernel 함수 개별 호출을 안내하지만, src/commands/의 executeStart 등 command 함수가 이미 동일 오케스트레이션을 수행하고 src/index.ts에서 export되고 있음. 프로토콜 문서와 공개 API의 정합성 검증이 필요한 상태 (출처: dispatch script 설계 리뷰, 2026-03-16)
- [사실] sprint-kit의 기존 소비 파이프라인(`buildOntologyIndex`)은 glossary/actions/transitions 3개 YAML만 입력받습니다. `domain_flows` 같은 새 카테고리를 추가하려면 소비 파이프라인의 스키마 확장이 선행되어야 합니다. 생성 파이프라인의 산출물 범위가 소비 파이프라인의 입력 범위를 초과하면 초과분은 버려집니다. (출처: ontology-auto-generation 설계 리뷰, 2026-03-23)
