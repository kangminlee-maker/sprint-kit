# onto_semantics — sprint-kit 도메인 학습

- "온톨로지를 적용한다"는 제안 검토 시, 해당 온톨로지의 현재 역할(입력/분류/출력)을 먼저 식별. 역할이 다른 개념을 합치면 관심사 분리 파괴 (출처: 산출물 온톨로지 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- "정보 구조와 렌더링 분리" 준수 여부는 "별도 계층 파일 존재"가 아니라 "구조화된 정보와 표현 로직이 분리되어 있는가"로 판정 (출처: 산출물 온톨로지 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- "X-aware"를 시스템 정체성에 사용할 때, "awareness의 범위"를 명시하지 않으면 "이 범위까지는 aware하지 않잖아"라는 질문이 반드시 발생. 정체성 용어에는 범위 한정이 동반되어야 함 (출처: 설계철학 검토, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- [사실] scripts/ 디렉토리의 "읽기 전용" 규칙은 동형이의어 문제를 가짐. 의미 A(appendScopeEvent 비호출)와 의미 B(파일시스템 비변경) 두 가지로 분열. 기존 4개 scripts는 의미 B에 해당하나, 규칙 문면이 의도를 부정확하게 축약하고 있음 (출처: dispatch script 설계 리뷰, 2026-03-16)
- invariant와 best-effort의 경계 기준: "시스템이 의미 해석 없이 구조만으로 탐지 가능한가". 구조적 탐지 가능 → invariant(코드), 의미 해석 필요 → best-effort(프로토콜) (출처: 설계철학 검토, 2026-03-11) (→ 글로벌 승격 완료, 2026-03-15)
- "판단 도구"와 "구현 템플릿"의 구분 기준은 독자의 행동. 판단 도구의 독자는 결정하고, 구현 템플릿의 독자는 코드로 옮김 (출처: Surface 생성 설계 과제 리뷰, 2026-03-10) (→ 글로벌 승격 완료, 2026-03-15)
- "LLM-ready" 산출물 확장의 비즈니스 가치는 직접 소비자(Builder/AI)의 품질 향상이지, PO의 판단력 향상이 아님. 이 간접 경로를 명시하지 않으면 우선순위 판단이 어려워짐 (출처: 10건 피드백 구현 계획 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- 외부 시스템 제한(Figma timestamp)으로 인한 데이터 품질 비대칭을 문서에 명시하지 않으면, 소비자가 모든 소스의 정밀도가 동일하다고 가정 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- "계약 문서"와 "가이드 문서" 분리가 유지보수 비용의 핵심 결정. 계약=코드 1:1 대응, 가이드=의도 기술 (출처: blueprint 온톨로지 설계 리뷰, 2026-03-09) (→ 글로벌 승격 완료, 2026-03-15)
- Sprint Kit의 온톨로지는 구조적 지식(엔티티, 필드, 관계)과 행동적 지식(전이 규칙, Gate Guard 조건, Compile Defense 로직)으로 구성되며, 후자가 시스템의 정체성을 결정. 스키마는 구조적 지식만 담을 수 있으므로 "Schema = Ontology"는 성립하지 않음 (출처: Schema-as-Ontology 리뷰, 2026-03-14)
- `ConstraintDecision`의 `modify-direction` 값은 하이픈(`-`)을 사용하며, 다른 enum 값들의 밑줄(`_`) 관례와 불일치. 문서 작성 시 관례 기반 추측(`_`)으로 대체될 위험이 높음 (출처: Schema-as-Ontology 리뷰, 2026-03-14)
- types.ts의 열거형은 모두 `as const` 배열 + union type 또는 직접 union type 선언으로 구현됨. TypeScript `enum` 키워드는 사용되지 않음. 문서나 생성 스크립트에서 "enum"이라 지칭하면 실제 구현과 불일치 (출처: ontology-map 리뷰, 2026-03-14)
- 타입의 물리적 위치(파일)가 의미를 내포함. compile-defense.ts에 위치 = "compile 문맥 전용", kernel/types.ts에 위치 = "시스템 전체 계약". 승격/이동 시 이 의미 변경을 의도적으로 결정해야 함 (출처: ontology-map 리뷰, 2026-03-14)

- [사실] ImportedSymbol.source 필드는 TypeScript에서 '파일 상대 경로', Go에서 '패키지 경로', Java에서 '패키지 FQN'으로, 단일 필드에 4가지 의미가 공존하는 동형이의어입니다. 호출 그래프 빌더가 이 값을 파일 경로로 해석하는 로직은 TypeScript 전용이며, 다언어 확장 시 해석 전략 분기가 필요합니다. (출처: Stage 1 다언어 확장 설계 리뷰, 2026-03-26)
