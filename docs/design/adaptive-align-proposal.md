# Adaptive Align — 탐색적 문답 기반 요구사항 공동 설계

> 상태: 제안 (proposal)
> 작성일: 2026-03-12
> 관련: 6-Agent Panel Review 2건 (validation-plan 업그레이드, 코드 독립적 검증)

---

## 1. 현재 구조의 문제

### 현재 흐름

```
사용자가 brief 작성 → /start (grounding) → Align Packet 제시 → /align (승인/거부)
```

sprint-kit은 **사용자가 이미 정의한 요구사항을 검증하는 역할**입니다.

### 근본 문제: 입력 품질이 모든 산출물 품질을 결정

brief의 품질이 낮으면(추상적, 자기모순, brownfield 미반영), 이후 모든 산출물이 그 낮은 품질을 물려받습니다.

- **Align Packet**: 모호한 brief에서 추출된 모호한 방향 → PO가 정확한 판단을 내릴 수 없음
- **Surface**: 추상적 요구사항에서 생성된 UI → 에이전트의 자의적 해석이 개입
- **Build Spec**: 모순된 요구사항에서 도출된 구현 계획 → compile 후에야 충돌 발견
- **PRD**: 위 산출물을 통합한 문서 → 원본의 모호함이 그대로 전파

constraint 발견으로 문제를 사후 보고할 수 있지만, "사용자가 진짜 원하는 것이 무엇인가"를 함께 찾아주지 않습니다.

## 2. 제안: 탐색적 문답 (Adaptive Align)

### 핵심 아이디어

sprint-kit의 역할을 **"검증자"에서 "공동 설계자"로** 확장합니다.

사용자가 목적과 개략적 가설만 brief에 넣으면, sprint-kit이:

1. **brownfield를 기반으로 현재 서비스 상태를 설명**합니다 (현재 userflow, 기존 feature, 상태 분기)
2. **온톨로지를 기반으로 변경 옵션을 제시**합니다 (UX 전문가 관점에서 현재 서비스 구조를 고려한 선택지)
3. **사용자의 선택에 따라 요구사항을 점진적으로 구체화**합니다
4. **구체화된 요구사항으로 Align Packet을 생성**합니다

### 제안 흐름

```
목적+가설 → /start (grounding) → 탐색적 문답 (N회) → Align Packet → /align (승인)
```

### 이 방향이 해결하는 3가지 문제

#### 자기모순 제거

현재: 사용자가 brief에 "체험 수업은 무료 취소 가능"과 "노쇼 시 자동 소진"을 동시에 기입 → grounding 이후 constraint로 발견 → "어느 쪽을 포기하시겠습니까?"

제안: 문답 과정에서 "체험 수업 취소 정책을 어떻게 하시겠습니까?"라는 질문에 사용자가 답하면서 자연스럽게 하나의 일관된 정책이 형성됩니다. 모순이 발생할 수 없는 구조입니다.

#### brownfield 반영

현재: 사용자가 기존 서비스의 내부 구조를 모른 채 brief를 작성합니다. "홈 화면에 프로그레스바를 추가하고 싶다"고 쓰지만, 현재 홈 화면의 상태 분기가 4가지뿐이라는 사실을 모릅니다.

제안: sprint-kit이 먼저 "현재 홈 화면은 4가지 상태(미신청/대기/레슨 중/결제 유도)로 분기됩니다"라고 설명한 뒤, "어떤 상태를 추가하거나 변경하시겠습니까?"를 묻습니다. 사용자는 현실에 기반한 결정을 내립니다.

#### 요구사항 정밀도 향상

현재: brief의 "기대 결과"가 "각 단계에서 다음 행동을 명확히 유도"처럼 추상적.

제안: "레슨 완료 후 사용자에게 보여줄 화면에서 (A) 다음 예약을 바로 유도, (B) 복습 콘텐츠 먼저 제시 후 예약 유도, (C) 결제 페이지로 직접 이동 — 어느 방향이 적합합니까?"처럼 구체적 선택지를 거치면, 요구사항이 실행 가능한 수준까지 정밀해집니다.

## 3. 설계 구체화

### 현재 흐름에서의 위치

`grounding` 이후, `align.proposed` 이전에 **탐색적 문답 단계**가 추가됩니다.

```
현재:  brief 작성 → /start (grounding) → Align Packet → /align (승인)
제안:  목적+가설 → /start (grounding) → 탐색적 문답 (N회) → Align Packet → /align (승인)
```

### 문답 1회(round)의 구조

문답이 자유 대화가 되면 추적성(traceability)이 사라집니다. sprint-kit의 핵심 가치인 "모든 결정이 이벤트로 기록된다"를 유지하려면, 문답에도 구조가 필요합니다.

```
1. sprint-kit이 현재 상태를 설명 (brownfield 기반)
   → "현재 홈 화면의 상태 분기는 4가지입니다: ..."

2. 변경 방향 옵션 제시 (온톨로지 기반)
   → "이 구조를 변경하는 방향은 3가지가 있습니다:
      (A) 기존 4상태에 세분화 상태를 추가
      (B) 상태 판정 로직 자체를 재설계
      (C) 기존 구조 유지하되 UI 레벨에서만 분기"

3. 사용자 선택 + 추가 의견
   → "A로 가되, 노쇼 상태는 별도로 분리하고 싶어"

4. 선택 기록 (이벤트)
   → exploration.decision_recorded { round: 1, topic: "상태 분기", choice: "A", note: "..." }
```

### 온톨로지의 역할

현재 온톨로지는 grounding에서 constraint를 발견하는 용도로만 사용됩니다. 탐색적 문답에서는 온톨로지가 추가 역할을 합니다:

| 역할 | 설명 |
|------|------|
| 질문 생성 | 온톨로지에 정의된 엔티티(수강권, 레슨, 매칭 등)와 사용자 목적을 교차하여, "이 변경이 어떤 엔티티에 영향을 미치는가?"를 도출 |
| userflow 설명 | 온톨로지의 상태 전이 정의를 사용하여, "현재 사용자가 겪는 흐름"을 설명 |
| 옵션 생성 | 온톨로지의 비즈니스 규칙과 제약을 기반으로, 각 변경 방향이 어떤 규칙과 충돌하는지를 미리 계산하여 옵션에 포함 |

### grounding 결과의 확장

탐색적 문답을 지원하려면, grounding 결과가 constraint 목록뿐 아니라 **서비스 현황 설명**도 포함해야 합니다.

| 현재 grounding 결과 | 추가 필요한 결과 |
|---------------------|-----------------|
| constraint 목록 (N건) | 현재 userflow 서술 (엔티티별 상태 전이) |
| perspective_summary | 기존 feature 목록 (영향 범위 파악용) |
| source_hashes | 상태 분기 현황 (코드 분석 결과) |

### 상태 머신 관점

두 가지 접근이 가능합니다:

**접근 A: 상태 추가 없이 `grounded` 상태 내에서 처리 (권장)**

`grounded` 상태에서 관찰 이벤트(`exploration.round_completed` 등)를 반복 기록하면서 문답을 진행합니다. constraint discovery는 기존 메커니즘을 그대로 사용합니다. 문답이 충분히 진행되면 기존대로 `align.proposed`를 발행합니다.

장점: 상태 머신 변경 최소. 기존 테스트 영향 없음.

**접근 B: `exploring` 상태 추가**

`grounded` → `exploring` → `align_proposed`로 새 상태를 추가합니다.

장점: 문답 진행 중인 scope와 문답 완료된 scope를 상태로 구분 가능.
단점: 상태 머신 매트릭스 확장, 전이 규칙 추가 필요.

### 종료 조건

문답이 발산하지 않도록 종료 조건이 필요합니다:

- **미결정 토픽이 0개가 되면** Align Packet 생성을 제안합니다
- **5회 이상 반복되면** 현재까지의 결정을 정리하여 제시합니다
- **사용자가 "충분하다"고 판단하면** 즉시 Align Packet 생성으로 진행합니다

## 4. 주의사항

### 에이전트 프로토콜 vs 코드 강제의 경계

탐색적 문답은 본질적으로 **의미 해석**입니다. "어떤 옵션을 제시할 것인가", "사용자의 답변에서 어떤 결정을 추출할 것인가"는 코드로 자동화하기 어렵습니다. 따라서:

- **에이전트 프로토콜**: 문답 절차, 질문 생성 규칙, 옵션 제시 형식
- **코드**: 이벤트 기록, 종료 조건 검증, round 횟수 제한 (gate-guard)

### 기존 brief 호환

탐색적 문답을 도입해도, 이미 상세한 brief를 가진 사용자는 기존 흐름(brief → grounding → Align Packet)을 그대로 사용할 수 있어야 합니다. brief의 상세도에 따라 문답 횟수가 줄어드는 적응형(adaptive) 구조가 필요합니다.

### downstream 영향

탐색적 문답의 결과물은 기존 Align Packet과 동일한 형식이어야 합니다. downstream(draft, compile, validate)은 Align Packet만 소비하므로, 문답 과정이 어떠했든 최종 Align Packet이 기존 규격을 충족하면 변경이 없습니다.

## 6. 기존 설계 원칙과의 관계

- **"탐지는 코드, 판정은 에이전트"**: 탐색적 문답은 에이전트가 수행하는 의미적 활동입니다. 코드는 이벤트 기록과 구조적 검증만 담당합니다.
- **GC-010 "Compile은 새 결정을 하지 않는다"**: 탐색적 문답은 compile 이전에 완료됩니다. compile 단계에 영향을 주지 않습니다.
- **이벤트 소싱**: 각 문답 round의 결정이 이벤트로 기록되어 추적 가능합니다.
- **Surface는 판단 도구**: 탐색적 문답에서 구체화된 요구사항으로 Surface를 생성하면, Surface의 판단 도구 역할이 더 정확해집니다.
