이 문서는 Sprint Kit이 실제로 생산하는 핵심 산출물의 완성된 예시입니다.
가상 케이스 "튜터 차단 기능"을 사용합니다.

# Part 1. Align Packet

## Align Packet: 튜터 차단 기능

### 1. 당신이 요청한 것 (To-be)

**원문:** "튜터 차단 기능"

**시스템이 해석한 방향:**
학생이 특정 튜터를 차단하면, 향후 매칭에서 해당 튜터가 자동으로 제외됩니다.

**제안된 범위:**

| 포함 | 동의하시나요? |
|------|-------------|
| 차단 생성 (학생이 튜터를 차단) | |
| 차단 해제 (학생이 차단을 풂) | |
| 차단 목록 조회 (학생이 차단 현황 확인) | |
| 매칭 시 차단 반영 (차단된 튜터 자동 제외) | |

| 제외 | 동의하시나요? |
|------|-------------|
| 튜터에게 차단 사실 통보 | |
| 차단 사유 수집 | |
| 관리자 차단 기능 | |

**시나리오:**

> 시나리오 1: 학생 A가 수업 후 튜터 B를 차단합니다. 이후 수업 예약 시 튜터 B는 후보에 나타나지 않습니다.
>
> 시나리오 2: 학생 A가 차단 관리 화면에서 튜터 B의 차단을 해제합니다. 이후 튜터 B가 다시 매칭 후보에 포함됩니다.

---

### 2. 현재 현실 (As-is)

#### Experience 관점 — 지금 사용자가 보는 것

- 튜터 프로필 화면이 존재하지만, 차단 기능은 없습니다.
- 학생이 불쾌한 튜터를 다시 만나도, 할 수 있는 일이 없습니다.
- 설정 메뉴에 차단 관련 항목이 없습니다.

#### Policy 관점 — 지금 적용되는 규칙

- 이용약관에 "플랫폼은 학생과 튜터의 매칭을 공정하게 수행한다"는 조항이 있습니다.
- 데이터 보존 정책: 개인 데이터는 3년 후 삭제합니다.
- 운영 정책에 차단 남용 관련 규정은 아직 없습니다.

#### System 관점 — 지금 시스템이 할 수 있는 것

- 매칭 시스템은 언어, 시간대, 레벨만으로 튜터를 배정합니다. "이 튜터를 빼달라"고 요청하는 방법이 없습니다.
- 차단 정보를 저장할 공간(DB 테이블)이 아직 없습니다.
- 튜터 프로필 API는 존재하지만, 차단 관련 기능은 포함되어 있지 않습니다.

<details>
<summary>기술 상세 (Builder 참고용)</summary>

- 매칭 엔진: `matching-engine.ts`의 `findAvailableTutors()` → `rankByCompatibility()` → `assign()`. exclusion 파라미터 없음.
- DB: `tutor_block_list` 테이블 미존재.
- API: `/api/v1/tutors` — GET, POST. block 관련 endpoint 없음.
</details>

### 3. 충돌 지점 (Tension)

요청한 것(to-be)과 현재 현실(as-is) 사이에 3건의 충돌이 발견되었습니다.

| CST-ID | 관점 | 충돌 내용 | 처리하지 않으면 |
|--------|------|----------|---------------|
| CST-001 | System | 매칭 시스템에 특정 튜터를 제외하는 기능이 없습니다. 새로 만들어야 합니다. | 차단 버튼을 눌러도 실제 매칭에 반영되지 않습니다. |
| CST-002 | Policy | 이용약관 "공정 매칭" 조항과 차단 기능이 충돌할 수 있습니다. | 법적 이슈가 발생할 수 있습니다. |
| CST-003 | Experience | 현재 튜터 프로필 화면의 공간이 제한적입니다. 차단 버튼 추가 시 레이아웃 조정이 필요합니다. | 버튼이 다른 요소와 겹쳐 오터치(잘못 누름) 위험이 있습니다. |

**변경 규모:** 매칭 시스템 수정이 포함된 중간 규모 변경입니다. 화면 변경만이 아닌, 시스템 내부 로직 변경이 필요합니다.

### 4. 지금 결정할 것

1. 위 범위(포함/제외)에 동의하십니까?
2. 이 3건의 충돌을 인지한 상태에서 이 방향으로 진행하겠습니까?
3. 열린 질문(차단 한도, 즉시 적용 여부)은 Draft 단계에서 결정해도 됩니까?

선택: **Approve** / **Revise** / **Reject** / **Redirect**

# Part 2. Draft Packet

*Align에서 Approve 후, 시스템이 stateful mockup을 생성합니다. 사용자가 mockup을 반복 수정하여 "이것이 내가 원하는 모습"이라고 확정한 뒤, 시스템이 확정된 mockup에 대해 deep constraint discovery를 수행한 결과입니다.*

## Draft Packet: 튜터 차단 기능

### 1. 확정된 Surface

- **scope type**: experience
- **surface location**: `scopes/SC-2026-001/surface/preview/`
- **실행 방법**: `cd surface/preview && npm run dev`
- **mockup 반복**: 3회 수정 후 사용자 확정 (revision 4)

**시나리오 가이드** (mockup에서 직접 확인한 것):

| 시나리오 | 시작 | 동작 순서 | 확인된 동작 |
|---------|------|----------|------------|
| 차단 생성 | 튜터 프로필 | 차단 버튼 → 확인 모달 → 확인 | 성공 메시지, 버튼이 "차단됨"으로 변경 |
| 차단 해제 | 설정 → 차단 관리 | 해제 버튼 → 확인 → 확인 | 목록에서 제거 |
| 중복 차단 | 이미 차단된 튜터 | 차단 버튼 | "차단됨" 비활성 상태 |
| 매칭 제외 | 수업 예약 | 매칭 요청 | 차단된 튜터 미포함 |

---

### 2. 현재까지의 결정 현황

- Align에서 결정: 방향 승인, 범위 확정, CST-001~003 인지 완료
- Draft에서 결정 필요: 아래 8건 (0건 결정 완료, 8건 미결정)
- 잠금 전 필수 해소: `clarify` 상태 항목이 있으면 잠글 수 없음

### 3. 발견된 Constraint — 확정된 mockup 기준

시스템이 확정된 mockup에 대해 3개 관점으로 deep discovery를 수행한 결과입니다.
Align에서 인지한 3건 + 신규 5건 = 총 8건.

#### 결정 필수 (이것 없이는 기능이 작동하지 않음)

| CST-ID | 관점 | 내용 | 처리하지 않으면 | 선택지 | 권장 | 선택 |
|--------|------|------|---------------|--------|------|------|
| CST-001 | System | 매칭 시스템에 튜터 제외 기능 없음 | 차단해도 매칭에 반영 안 됨 | inject (추가) | **inject** | |
| CST-005 | System | 차단 정보 저장할 DB 테이블 없음 | 차단 기록을 저장할 수 없음 | inject (생성) | **inject** | |
| CST-008 | Policy | 차단 남용 방지 정책 없음. 한도 필요 | 학생이 전체 튜터를 차단할 수 있음 | inject (언어별 5명) / inject (전체 10명) / inject (제한 없음) | **inject (언어별 5명)** | |

#### 결정 권장 (기능은 작동하나 리스크 있음)

| CST-ID | 관점 | 내용 | 처리하지 않으면 | 선택지 | 권장 | 선택 |
|--------|------|------|---------------|--------|------|------|
| CST-002 | Policy | 이용약관 "공정 매칭" 조항 충돌 가능 | 법적 이슈 가능성 | inject (프레이밍 변경) / override (무시) / clarify (법무 확인) | **clarify** | |
| CST-003 | Experience | 프로필 화면 레이아웃 공간 부족 | 오터치 위험 | inject (간격 확보) / defer (현재 위치) | **inject** | |
| CST-004 | Experience | 차단 관리 메뉴가 설정의 4번째 항목 | 사용자가 찾기 어려울 수 있음 | inject (순서 변경) / defer (현재 위치) | **defer** | |
| CST-006 | System | 매칭 API 응답에 차단 정보 추가 시 기존 응답 형태 변경 | 기존 앱의 응답 파싱이 깨질 수 있음 | inject (optional 필드) / inject (별도 endpoint) | **inject (optional 필드)** | |
| CST-007 | Policy | 차단 기록은 개인 데이터 — 3년 후 삭제 필요 | 데이터 보존 정책 위반 | inject (삭제 로직) / defer (나중에) | **inject** | |

---

### 4. 제약 조건 (구현 시 반드시 지켜야 할 것)

이 항목들은 결정이 아니라 사실입니다. Align에서 잠긴 방향과 확정된 mockup에서 도출됩니다.

- 기존 예약된 수업은 취소하지 않는다 (차단은 향후 매칭에만 적용)
- 차단 행위는 중복 요청 시 오류 없이 기존 상태를 유지한다
- 튜터에게 차단 사실을 알리지 않는다 (이번 범위)
- 기존 API 응답 형식을 깨지 않는다

---

### 5. 리스크

| 리스크 | 영향 | 되돌림 비용 |
|--------|------|------------|
| 차단 한도가 낮으면 사용자 불만 | "더 이상 차단할 수 없다"에 좌절 | 낮음 — 한도 숫자만 변경하면 됨 |
| 매칭 풀 축소로 매칭 실패 증가 | 차단이 많은 학생은 매칭 가능 튜터가 줄어듦 | 낮음 — 안내 메시지 추가 |
| 이용약관 충돌 | 법적 분쟁 가능 | 높음 — 약관 변경 필요 |

# Part 3. 이 예시가 보여주는 것

## Align Packet의 역할

- **To-be → As-is → Tension** 구조로, 사용자가 자기 요청의 실제 윤곽을 봅니다.
- 범위 포함/제외를 시스템이 제안하되, **각각에 대해 사용자 동의를 요청**합니다.
- 기술 상세는 접혀 있고, 비즈니스 영향 중심으로 서술합니다.

## Draft Packet의 역할

- 사용자가 **직접 mockup을 반복 수정**하여 to-be를 Experience 언어로 정의합니다.
- 확정된 mockup에 대해 시스템이 **deep constraint discovery**를 수행합니다.
- constraint가 **"결정 필수"와 "결정 권장"**으로 분리되어 판단 부담이 줄어듭니다.
- 각 constraint에 **"처리하지 않으면"**이 명시되어, 기술을 몰라도 심각도를 판단할 수 있습니다.
- **되돌림 비용**이 표시되어, 결정의 무게를 가늠할 수 있습니다.

## "Constraint-aware translation"의 실현

- Align에서 3건, Draft에서 8건의 제약이 시스템에 의해 발견되었습니다.
- mockup 반복 중에도 가벼운 제약 힌트가 제공되어, Gate 2에서 처음 듣는 제약이 최소화됩니다.
- 이 제약들이 없었다면, 이용약관 충돌(CST-002), 데이터 보존 의무(CST-007), API 호환성(CST-006) 등은 개발 중에야 발견되었을 것입니다.
- 시스템이 제약을 미리 보여줌으로써, 사용자의 판단이 더 정확해졌습니다.
