# Part 1. Align Packet

## Align Packet: 튜터 차단 기능

### 1. 당신이 요청한 것 (To-be)

**원문:** "튜터 차단 기능"

**시스템이 해석한 방향:**
학생이 특정 튜터를 차단하면, 향후 매칭에서 해당 튜터가 자동으로 제외됩니다.

**제안된 범위:**

| 포함 | 동의하시나요? |
|------|-------------|
| 차단 생성 (학생이 튜터를 차단) | |
| 차단 해제 (학생이 차단을 풂) | |
| 차단 목록 조회 (학생이 차단 현황 확인) | |
| 매칭 시 차단 반영 (차단된 튜터 자동 제외) | |

| 제외 | 동의하시나요? |
|------|-------------|
| 튜터에게 차단 사실 통보 | |
| 차단 사유 수집 | |
| 관리자 차단 기능 | |

**시나리오:**

> 시나리오 1: 학생 A가 수업 후 튜터 B를 차단합니다. 이후 수업 예약 시 튜터 B는 후보에 나타나지 않습니다.
>
> 시나리오 2: 학생 A가 차단 관리 화면에서 튜터 B의 차단을 해제합니다. 이후 튜터 B가 다시 매칭 후보에 포함됩니다.

### 2. 현재 현실 (As-is)

#### Experience 관점 — 지금 사용자가 보는 것

- 튜터 프로필 화면이 존재하지만, 차단 기능은 없습니다.
- 학생이 불쾌한 튜터를 다시 만나도, 할 수 있는 일이 없습니다.
- 설정 메뉴에 차단 관련 항목이 없습니다.

#### Policy 관점 — 지금 적용되는 규칙

- 이용약관에 "플랫폼은 학생과 튜터의 매칭을 공정하게 수행한다"는 조항이 있습니다.
- 데이터 보존 정책: 개인 데이터는 3년 후 삭제합니다.
- 운영 정책에 차단 남용 관련 규정은 아직 없습니다.

#### Code 관점 — 지금 시스템이 할 수 있는 것

- 매칭 시스템은 언어, 시간대, 레벨만으로 튜터를 배정합니다. "이 튜터를 빼달라"고 요청하는 방법이 없습니다.
- 차단 정보를 저장할 공간(DB 테이블)이 아직 없습니다.
- 튜터 프로필 API는 존재하지만, 차단 관련 기능은 포함되어 있지 않습니다.

<details>
<summary>기술 상세 (Builder 참고용)</summary>

- 매칭 엔진: `matching-engine.ts`의 `findAvailableTutors()` → `rankByCompatibility()` → `assign()`. exclusion 파라미터 없음.
- DB: `tutor_block_list` 테이블 미존재.
- API: `/api/v1/tutors` — GET, POST. block 관련 endpoint 없음.

### 3. 충돌 지점 (Tension)

요청한 것(to-be)과 현재 현실(as-is) 사이에 3건의 충돌이 발견되었습니다.

| CST-ID | 관점 | 요약 |
|--------|------|------|
| CST-001 | Code | 매칭 시스템에 튜터 제외 기능이 없음 |
| CST-002 | Policy | 이용약관 "공정 매칭" 조항과 충돌 가능 |
| CST-003 | Experience | 튜터 프로필 화면에 차단 버튼 공간 부족 |

---

#### CST-001 | Code | 매칭 시스템에 튜터 제외 기능이 없음

**이것이 무엇인가:**
현재 매칭 시스템은 학생의 언어, 시간대, 레벨만으로 튜터를 배정합니다. "이 튜터를 후보에서 빼달라"는 요청을 받을 수 있는 구조가 없습니다. 차단 기능이 실제로 작동하려면, 매칭 과정에 "제외 목록"을 전달하는 기능을 새로 만들어야 합니다.

**왜 충돌하는가:**
차단 버튼을 화면에 추가하는 것만으로는 부족합니다. 버튼을 눌러서 차단 기록을 저장하더라도, 매칭 시스템이 그 기록을 읽지 않으면 다음 매칭에서 차단된 튜터가 다시 배정됩니다. 화면(Experience)과 시스템 내부(Code)가 함께 바뀌어야 차단이 실제로 작동합니다.

**처리하지 않으면:**
차단 버튼을 눌러도 실제 매칭에 반영되지 않습니다. 학생은 차단했다고 생각하지만, 차단된 튜터가 다시 배정됩니다.

**변경 규모:** 화면 변경만이 아닌, 매칭 시스템 내부 로직 변경이 필요합니다. 이번 범위에서 가장 큰 변경입니다.

<details>
<summary>기술 상세 (Builder 참고용)</summary>

매칭 엔진 `matching-engine.ts`의 `findAvailableTutors()` 함수에 exclusion 파라미터가 없습니다. 이 함수의 인터페이스를 변경하여 차단 목록을 입력으로 받을 수 있게 해야 합니다. 하위 호환을 유지하면서 변경해야 기존 매칭 동작에 영향이 없습니다.
</details>

---

#### CST-002 | Policy | 이용약관 "공정 매칭" 조항과 충돌 가능

**이것이 무엇인가:**
이용약관 제12조에 "플랫폼은 학생의 학습 조건(언어, 시간대, 레벨)에 따라 공정하게 튜터를 배정한다"고 명시되어 있습니다. 이 조항의 취지는 특정 튜터가 부당하게 배제되지 않도록 보호하는 것입니다.

**왜 충돌하는가:**
차단 기능은 학생이 특정 튜터를 매칭에서 제외하는 것입니다. 이것이 "공정한 배정"이라는 약관의 취지와 충돌합니다. 학생의 차단이 합리적 사유 없이 남용되면, 특정 튜터가 구조적으로 불이익을 받을 수 있습니다. 예를 들어 한 튜터가 여러 학생에게 차단되면, 해당 튜터의 수업 기회가 줄어듭니다.

**처리하지 않으면:**
차단 기능 출시 후 튜터 측에서 "약관 위반"을 주장할 근거가 생깁니다. 실제 분쟁이 발생하면 법무 검토 없이는 대응이 어렵습니다.

**변경 규모:** 약관 검토 또는 변경이 필요합니다. 기능 구현과 별도로 법무 절차가 수반됩니다.

**선택지:**
- 이용약관에 "학생의 선호 설정은 공정 매칭의 예외로 허용한다"는 단서를 추가 → 법무 검토 필요
- 차단 사유를 필수로 수집하여 "합리적 사유 기반 차단"으로 제한 → 이번 범위 확대 (현재 범위에서는 사유 수집을 제외함)
- 이 충돌을 인지하고 현재 약관 그대로 진행 → 리스크 감수
- 이 충돌에 대해 추가 확인이 필요 → Draft 단계에서 clarify로 처리

<details>
<summary>관련 약관 원문</summary>

이용약관 제12조 (매칭 서비스):
"플랫폼은 학생이 등록한 학습 조건(학습 언어, 희망 시간대, 학습 레벨)에 따라 적합한 튜터를 공정하게 배정한다. 플랫폼은 특정 튜터를 부당하게 우대하거나 배제하지 않는다."
</details>

#### CST-003 | Experience | 튜터 프로필 화면에 차단 버튼 공간 부족

**이것이 무엇인가:**
현재 튜터 프로필 화면에는 "즐겨찾기" 버튼, "메시지 보내기" 버튼, 튜터 정보 영역이 배치되어 있습니다. 디자인 가이드에 따르면 버튼 간 최소 간격은 16px이며, 현재 레이아웃에서 새 버튼을 추가할 여유 공간이 8px밖에 없습니다.

**왜 충돌하는가:**
차단 버튼을 현재 레이아웃에 그대로 추가하면, "즐겨찾기" 버튼과의 간격이 디자인 가이드 기준(16px)보다 좁아집니다. 두 버튼이 가까우면 사용자가 즐겨찾기를 누르려다 차단을 누르거나, 반대로 차단을 누르려다 즐겨찾기를 누르는 오터치가 발생할 수 있습니다. 차단은 되돌릴 수 있지만, 의도하지 않은 차단은 사용자 경험을 해칩니다.

**처리하지 않으면:**
오터치 위험이 있는 상태로 출시됩니다. 차단 버튼과 즐겨찾기 버튼을 혼동하는 사용자 문의가 발생할 수 있습니다.

**변경 규모:** 프로필 화면 레이아웃 재배치. 기존 화면 구조 변경이지만, 새 기능 추가는 아닙니다.

**선택지:**
- 프로필 화면 하단에 버튼 영역을 재배치하여 간격 확보 → 기존 화면 레이아웃 변경
- 차단 버튼을 "더보기(⋯)" 메뉴 안에 넣어 직접 노출하지 않음 → 차단 접근성 저하
- 즐겨찾기 버튼을 아이콘으로 축소하여 공간 확보 → 즐겨찾기 기능의 가시성 저하

<details>
<summary>현재 레이아웃 참고 (Figma)</summary>

현재 프로필 화면 상단 버튼 영역:
`[★ 즐겨찾기] [✉ 메시지]` — 두 버튼이 나란히 배치.
여유 공간: 우측 8px. 디자인 가이드 최소 간격: 16px.
Figma 위치: `figma.com/design/xxxxx` — Tutor Profile 프레임.
</details>

**변경 규모:** 매칭 시스템 수정이 포함된 중간 규모 변경입니다. 화면 변경만이 아닌, 시스템 내부 로직 변경과 정책 검토가 필요합니다.

#### CST-005 | Code | 차단 정보 저장 공간 없음 — 필수 (Builder 결정 항목)

**상황:** "누가 누구를 차단했는지" 기록을 보관할 DB 테이블(데이터를 행과 열로 보관하는 저장 공간)이 없습니다.

**처리하지 않으면:** 차단 버튼을 눌러도 기록이 남지 않아, 앱을 다시 열면 차단이 풀려 있습니다.

**Builder가 결정할 사항:** 차단 기록을 저장할 DB 테이블을 새로 만들어야 합니다. 테이블의 구조(어떤 열을 둘지, 검색 속도를 위한 인덱스(index — 데이터를 빠르게 찾기 위한 목차) 설정 등)는 Builder가 결정합니다.

**이 작업 관점에서의 판단:** 저장 공간이 없으면 차단 기능 전체가 성립하지 않으므로 반드시 만들어야 합니다. 되돌림 비용은 낮습니다 (테이블을 삭제하면 원래대로). 구현 방식은 Builder의 재량이며, 제품 판단이 필요한 사항은 아닙니다.

#### CST-008 | Policy | 차단 남용 방지 한도 없음 — 필수

**상황:** 현재 차단 횟수에 대한 제한이 없습니다. 한도가 없으면 학생이 모든 튜터를 차단할 수 있고, 그러면 매칭 가능한 튜터가 없어져 수업 예약 자체가 불가능해집니다.

**처리하지 않으면:** 학생이 전체 튜터를 차단할 수 있습니다. 매칭 실패가 빈번해지고, 운영 문의가 증가합니다.

**선택지:**

| 선택 | 내용 | 리스크 | 되돌림 비용 |
|------|------|--------|------------|
| inject (언어별 5명) | 학생이 학습 중인 언어별로 최대 5명까지 차단 가능 | 한도가 낮아서 "더 이상 차단할 수 없다"는 불만 가능 | 낮음 — 숫자만 변경 |
| inject (전체 10명) | 언어 구분 없이 전체 10명까지 차단 가능 | 특정 언어의 튜터를 집중 차단하면 해당 언어 매칭 풀이 급격히 줄어듦 | 낮음 — 숫자만 변경 |
| inject (제한 없음) | 차단 횟수 제한 없음 | 전체 튜터를 차단할 수 있어 매칭 자체가 불가능해질 수 있음 | 낮음 — 한도 추가하면 됨 |

추천: **inject (언어별 5명)** — 남용을 방지하면서 학생의 선택권을 보장합니다.

#### CST-002 | Policy | 이용약관 "공정 매칭" 조항 충돌 — 권장

**상황:** 이용약관 제12조에 "플랫폼은 학생의 학습 조건에 따라 공정하게 튜터를 배정한다. 특정 튜터를 부당하게 배제하지 않는다"고 명시되어 있습니다. 차단 기능은 학생이 특정 튜터를 매칭에서 제외하는 것이므로, 이 조항의 취지와 충돌합니다. 한 튜터가 여러 학생에게 차단되면 수업 기회가 구조적으로 줄어들 수 있습니다.

**처리하지 않으면:** 차단 기능 출시 후 튜터 측에서 "약관 위반"을 주장할 근거가 생깁니다. 실제 분쟁 시 법무 검토 없이는 대응이 어렵습니다.

**선택지:**

| 선택 | 내용 | 리스크 | 되돌림 비용 |
|------|------|--------|------------|
| inject (예외 조항 추가) | 이용약관에 "학생의 선호 설정은 공정 매칭의 예외로 허용한다"는 단서를 추가 | 법무 검토 비용 발생. 약관 변경에 시간 소요 | 중간 — 약관 재변경 필요 |
| override (현재 약관 유지) | 충돌을 인지하고 현재 약관 그대로 진행 | 튜터 측 항의 시 법적 대응 근거 약함 | 높음 — 사후 약관 변경 + 이미 발생한 분쟁 대응 |
| clarify (법무 확인) | 법무팀에 확인 후 결정. 확인 완료 전까지 이 항목은 미결정 상태 | 확인에 시간이 걸리면 전체 일정 지연 | — |

추천: **clarify** — 법적 리스크가 있는 사안이므로 법무팀 확인 후 결정하는 것이 안전합니다.

#### CST-003 | Experience | 프로필 화면 차단 버튼 공간 부족 — 권장

**상황:** 현재 튜터 프로필 화면에 "즐겨찾기"와 "메시지" 버튼이 나란히 배치되어 있습니다. 디자인 가이드에 따르면 버튼 간 최소 간격은 16px인데, 새 버튼을 추가할 여유 공간이 8px뿐입니다. 두 버튼이 가까우면 즐겨찾기를 누르려다 차단을 누르거나, 반대의 오터치가 발생합니다.

**처리하지 않으면:** 오터치 위험이 있는 상태로 출시됩니다. 의도하지 않은 차단에 대한 사용자 문의가 발생할 수 있습니다.

**선택지:**

| 선택 | 내용 | 리스크 | 되돌림 비용 |
|------|------|--------|------------|
| inject (간격 확보) | 버튼 영역을 재배치하여 디자인 가이드 기준(16px) 충족 | 기존 화면 레이아웃이 바뀌어 사용자가 잠시 혼동할 수 있음 | 낮음 — 레이아웃 재조정 |
| defer (현재 위치 유지) | 차단 버튼을 "더보기(⋯)" 메뉴 안에 넣어 공간 문제를 회피 | 차단 기능의 접근성이 떨어짐. 사용자가 차단 방법을 찾지 못할 수 있음 | 낮음 — 버튼 위치 이동 |

추천: **inject** — 오터치 방지가 사용자 경험에 직접 영향을 줍니다.

#### CST-004 | Experience | 차단 관리 메뉴 위치 — 권장

**상황:** 차단 관리 화면은 설정 메뉴의 4번째 항목에 위치합니다. 자주 사용하는 기능(프로필 수정, 알림 설정)이 위에 있고, 차단 관리는 스크롤해야 보입니다.

**처리하지 않으면:** 차단 관리 화면을 찾기 어려울 수 있습니다. 다만 차단 관리는 자주 사용하는 기능이 아니므로, 즉각적인 문제는 크지 않습니다.

**선택지:**

| 선택 | 내용 | 리스크 | 되돌림 비용 |
|------|------|--------|------------|
| inject (순서 변경) | 차단 관리를 설정 메뉴 2번째로 올림 | 기존 설정 메뉴 순서에 익숙한 사용자가 혼동할 수 있음 | 낮음 — 순서 재변경 |
| defer (현재 위치 유지) | 4번째 항목에 그대로 둠 | 찾기 어려울 수 있으나, 차단 관리 자체가 드문 행위이므로 영향 작음 | — |

추천: **defer** — 차단 관리는 자주 쓰는 기능이 아니므로, 현재 위치로도 충분합니다. 사용 데이터를 본 뒤 조정해도 됩니다.

#### CST-006 | Code | 매칭 API 응답 형태 변경 위험 — 권장 (Builder 결정 항목)

**상황:** 매칭 API(매칭 결과를 앱에 전달하는 통신 규격)의 응답에 차단 정보를 포함하면, 기존 모바일 앱이 이 응답을 읽는 방식이 달라질 수 있습니다. 현재 앱은 "튜터 이름, 시간, 레벨"만 읽도록 만들어져 있는데, 여기에 "차단 여부"가 추가되면 기존 앱이 이 새 항목을 처리하지 못할 수 있습니다.

**처리하지 않으면:** 기존 모바일 앱이 매칭 결과를 제대로 표시하지 못할 수 있습니다. 앱 업데이트가 안 된 사용자에게 오류가 발생할 수 있습니다.

**Builder가 결정할 사항:** 차단 정보를 기존 API에 어떻게 포함할지는 Builder가 결정합니다. 두 가지 방식이 있습니다.

- **optional 필드 방식** — 기존 응답은 그대로 유지하고, optional 필드(선택적 추가 항목)로 차단 정보를 붙입니다. 기존 앱은 이 항목을 무시하므로 영향 없습니다. 구현 비용이 적지만, 하나의 API에 역할이 섞일 수 있습니다.
- **별도 endpoint 방식** — 별도 endpoint(차단 전용 조회 주소)를 새로 만듭니다. 기존 매칭 API는 전혀 건드리지 않습니다. 기존 앱에 영향이 완전히 없지만, 구현 비용이 더 들고 유지보수할 API가 하나 늘어납니다.

**이 작업 관점에서의 판단:** 어떤 방식이든 기존 앱에 영향을 주지 않아야 한다는 것이 핵심입니다 (제약 조건 "기존 API 응답 형식을 깨지 않는다" 참조). optional 필드 방식이 구현 비용이 적고 이 제약을 만족하므로 적합합니다. 최종 구현 방식은 Builder가 코드 구조를 보고 결정합니다.

# Part 3. 이 예시가 보여주는 것

## Align Packet의 역할

- **To-be → As-is → Tension** 구조로, 사용자가 자기 요청의 실제 윤곽을 봅니다.
- 범위 포함/제외를 시스템이 제안하되, **각각에 대해 사용자 동의를 요청**합니다.
- 기술 상세는 접혀 있고, 비즈니스 영향 중심으로 서술합니다.

## Draft Packet의 역할

- 사용자가 **직접 mockup을 반복 수정**하여 to-be를 Experience 언어로 정의합니다.
- 확정된 mockup에 대해 시스템이 **deep constraint discovery**를 수행합니다.
- constraint가 **"결정 필수"와 "결정 권장"**으로 분리되어 판단 부담이 줄어듭니다.
- 각 constraint에 **"처리하지 않으면"**이 명시되어, 기술을 몰라도 심각도를 판단할 수 있습니다.
- **되돌림 비용**이 표시되어, 결정의 무게를 가늠할 수 있습니다.

## "Constraint-aware translation"의 실현

- Align에서 3건, Draft에서 8건의 제약이 시스템에 의해 발견되었습니다.
- mockup 반복 중에도 가벼운 제약 힌트가 제공되어, Gate 2에서 처음 듣는 제약이 최소화됩니다.
- 이 제약들이 없었다면, 이용약관 충돌(CST-002), 데이터 보존 의무(CST-007), API 호환성(CST-006) 등은 개발 중에야 발견되었을 것입니다.
- 시스템이 제약을 미리 보여줌으로써, 사용자의 판단이 더 정확해졌습니다.
