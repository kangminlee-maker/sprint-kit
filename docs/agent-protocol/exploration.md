# Exploration Protocol

grounding 완료 후, Align Packet 생성 전에 PO와 함께 요구사항을 탐색적으로 구체화하는 프로토콜입니다.
설계 근거: `docs/design/adaptive-align-conversation-design.md`

## 진입 조건

- 현재 상태가 `grounded`
- grounding.completed 이벤트가 기록됨

## 진입 판별

brief의 상세도에 따라 exploration 범위가 달라집니다.

| 진입 방식 | 판별 기준 | exploration 범위 |
|-----------|----------|----------------|
| **대화만** | brief.md가 없거나 필수 항목이 모두 비어 있음 | 6개 Phase 전체 수행 |
| **간략 brief** | 변경 목적만 채워져 있고 나머지(대상 사용자, 기대 결과, 포함/제외)가 비어 있음 | Phase 1에서 목적 확인 후 Phase 2~6 수행 |
| **상세 brief** | 모든 필수 항목이 채워져 있음 | Phase 1에서 목적 재정위 후, brief에 이미 결정된 부분은 확인만 하고 건너뜀 |

진입 시 `exploration.started` 이벤트를 기록합니다:

```typescript
appendScopeEvent(paths, {
  type: "exploration.started",
  actor: "agent",
  payload: {
    entry_mode: "conversation" | "brief_minimal" | "brief_detailed",
    initial_goals: ["PO가 최초에 언급한 목표 목록"],
  },
});
```

## 6개 Phase

```
Phase 1: 목적 정밀화     — "달성하려는 결과는 무엇인가?"
Phase 2: 영역 탐색       — "이 결과에 어떤 영역이 관련되는가?"
Phase 3: 현재 상태 공유   — "각 영역이 지금 어떻게 동작하는가?"
Phase 4: 시나리오 탐색    — "각 영역에서 사용자가 어떤 경험을 하길 원하는가?"
Phase 5: 가정 검증       — "이 방향에 어떤 가정이 깔려 있는가?"
Phase 6: 범위 확정       — "이번에 바꾸는 것과 바꾸지 않는 것"
```

---

### Phase 1: 목적 정밀화

PO가 어떤 수준에서 시작하든, **"달성하려는 결과(outcome)"** 수준으로 재정위합니다.

**재정위 방법**:

| PO의 시작점 | 재정위 |
|-----------|--------|
| 목표(outcome) | 이미 목표 수준. 구체화 질문 |
| 해결책(solution) | "그것으로 달성하려는 결과는?" |
| 문제(pain point) | "이 문제를 줄이면 달성하려는 결과는?" |
| 기능 명세(feature) | "이 기능들이 달성하려는 결과를 먼저 확인" |
| 지표(metric) | "이탈 지점을 파악하면 어디서 확보할 수 있는지 알 수 있습니다" |

**규칙**:
- PO의 시작점을 부정하지 않음 ("그것은 목표가 아닙니다" → 금지)
- 결과(outcome)는 "사용자 또는 비즈니스의 상태 변화"로 정의
- PO가 제시한 해결책(기능)은 인정하되 보류: "Phase 4에서 대안과 함께 검토합니다"
- 해결 방향을 제안하지 않음

**전환 기준**: PO가 결과 목록과 우선순위에 동의하면 Phase 2로 이동.

---

### Phase 2: 영역 탐색

목표에 **어떤 서비스 영역이 관련되는가**를 발견합니다.

**탐색 방법** (목표의 성격에 따라 선택):
1. 엔티티 기반 Top-down: 관련 엔티티 → 관여하는 화면/기능/정책
2. 퍼널 기반: 전환 단계별 이탈 지점 → 관련 영역
3. 터치포인트 매핑: 모든 사용자 접점 → 기여/방해 평가
4. 영향 체인 역추적: "이것이 달성되려면?" 재귀 추적

**ClickHouse 연동**: `.sprint-kit.yaml`에 ClickHouse MCP가 등록되어 있으면, 퍼널 전환율/상태별 분포를 조회하여 영역 우선순위를 데이터로 보강합니다.

**영역 제시 형식**:
```
| 영역 | 경로 위치 | 현재 격차 |
|------|----------|----------|
| 홈 화면 | 필수 경유지 | 격차 큼 — 해당 상태 없음 |
| 결제 플로우 | 필수 경유지 | 격차 작음 — 플로우 존재 |
```

**PO가 sprint-kit이 제시하지 않은 영역을 언급할 때**:
- "직접적인 인과 관계가 파악되지 않았습니다. 제가 놓친 부분이 있을 수 있으니, 관련된다고 판단하신 근거를 알려주세요."
- PO가 근거를 제시하면 함께 검토, 근거가 없으면 이번 scope에서 제외

**PO가 기존 자산/기능을 언급할 때**:
- brownfield에서 해당 기능의 구현 상태를 확인하여 보고
- brownfield에서 확인되지 않으면 PO에게 상세 요청

**전환 기준**: PO가 영역 목록에 동의하면 Phase 3으로 이동.

---

### Phase 3: 현재 상태 공유

각 영역의 **현재 동작을 PO가 이해할 수 있는 언어**로 설명합니다.

**번역 규칙**:
- PO에게 보여주는 것: "사용자가 보는 것"과 "사용자가 할 수 있는 것"
- PO에게 보여주지 않는 것: API, DB 스키마, 모듈 의존성 (PO가 질문하면 설명)

**앱 외부 채널 확인 (표준 질문)**:
현재 상태 설명 후 반드시 다음을 묻습니다:
> "위는 앱 안에서 학생이 보는 것입니다. 앱 밖에서 학생에게 전달되는 것이 있습니까? (예: 카카오 알림톡, 문자, 이메일 등)"

**규칙**:
- 변경 방향을 제안하지 않음 (Phase 4에서 시나리오를 통해 도출)
- 기술적 난이도를 언급하지 않음

**전환 기준**: PO가 현재 상태 설명(앱 내부 + 외부 채널)에 동의하면 Phase 4로 이동.

---

### Phase 4: 시나리오 탐색

각 영역에서 **사용자가 어떤 경험을 하길 원하는가**를 탐색합니다.

**시나리오 순서 결정 규칙**:
1. 1차 정렬 — Phase 1 결과 우선순위 (결과 A 시나리오 먼저)
2. 2차 정렬 — Phase 2 격차 크기 (격차 큼 먼저)
3. 3차 정렬 — 사용자 빈도 (빈번한 시나리오 먼저, 예외 나중)

**시나리오 1개당 흐름: 추출 → 제안 → 확정**

```
[Step 1 — 추출] PO의 의도를 먼저 확인
  "이 학생이 홈 화면에서 가장 먼저 인지해야 하는 것은 무엇입니까?"

[Step 2 — 제안] PO 답변 확인 후 대안 제시
  "확인했습니다. {PO 의도 요약}. 참고로 다른 접근도 있습니다:
  (A) 현재 방향 — {PO가 말한 것}
  (B) {brownfield/온톨로지/설계 패턴 기반 대안}
  현재 방향을 유지하시겠습니까?"

[Step 3 — 확정] 최종 선택 기록
```

**대안 제시 규칙**:
- PO가 답하기 전에 대안을 먼저 제시하지 않음 (유도 질문이 됨)
- 대안을 "더 좋은 것"으로 프레이밍하지 않음
- 기존 방향 + 대안 1~2개. 5개 이상 나열 금지
- PO가 유지를 선택하면 존중. 같은 방향에 대해 재고하게 만들지 않음

**대안 소스 3가지**:
- brownfield (기존 시스템): "현재 시스템에 이미 이런 기능이 있습니다"
- 온톨로지 (도메인 규칙): "이 도메인에서 일반적으로 사용되는 패턴입니다"
- 설계 패턴 (UX 일반): "이런 상황에서 효과적인 접근이 있습니다"

**규칙**:
- 시나리오당 질문 2~3개 제한
- PO의 답변을 해석한 후 "이렇게 이해했는데 맞습니까?" 확인
- 기술적 제약을 이유로 시나리오를 제한하지 않음

**전환 기준**: 핵심 시나리오를 모두 다루면 Phase 5로 이동.

---

### Phase 5: 가정 검증

Phase 1~4에서 축적된 결정에 깔린 **암묵적 가정**을 드러냅니다.

**가정의 3가지 유형**:
- 사용자 행동 가정: "체험 직후 결제 의향이 높다" → 틀리면 유도 시점이 무효
- 서비스 정책 가정: "튜터 노쇼 시 자동 복구" → 정책 변경이 필요할 수 있음
- 기술 전제 가정: "API로 상태 판별 가능" → 구현 범위에 영향

**정책 가정 교차 대조**: 온톨로지/이용약관 소스와 대조하여 confirmed/unverified 표시.

**기술 전제 가정의 보류 규칙**: "구현 단계에서 확인"으로 보류할 때, 반드시 **불가능 시 대안**을 함께 기록합니다:
> "가능하면: 시나리오 1 그대로. 불가능하면: 리포트 대신 상태 메시지로 대체. 대안이 Phase 1의 결과를 여전히 달성하면 Builder가 판단. 결과 자체가 변경되면 PO 확인 필요."

**모순 발견 시**: 즉시 PO에게 알리고 두 가정 중 하나를 수정하도록 요청.

**scope 변경 시 시나리오 역수정**:
1. 영향받는 Phase 4 시나리오를 식별
2. 변경 사항을 PO에게 즉시 안내
3. PO가 승인하면 Phase 6으로, 거부하면 Phase 4 재탐색

**전환 기준**: 모든 가정이 확인/수정되고, scope 변경 영향이 처리되면 Phase 6으로 이동.

---

### Phase 6: 범위 확정

Phase 1~5 결과를 종합하여 **포함/제외/확인된 가정/미확인 가정**을 확정합니다.

**제시 형식**:
```
[포함 — 이번에 바꾸는 것]
  ✓ {항목} (시나리오 N에서 도출)

[제외 — 이번에 바꾸지 않는 것]
  ✗ {항목} — {제외 이유}

[확인된 가정]
  • {가정} ({확인 소스})

[미확인 가정 — 구현 단계에서 확인]
  • {가정} (불가능 시 대안: {대안})
```

**전환 기준**: PO가 범위에 동의하면 exploration 완료. Align Packet 생성으로 진행.

## 대화 로그 기록

exploration 대화는 `build/exploration-log.md`에 실시간으로 기록합니다.

### 형식

```markdown
# Exploration Log — {scope-id}

## Phase 1: 목적 정밀화
- **상태**: 완료
- **확정된 결과**: ...

### Round 1
**sprint-kit**: {질문/정보 제시}
**PO**: {응답}
**결정**: {결정 요약}
```

### 기록 규칙

- **기록 시점**: 매 round 종료 시 append
- **append-only**: 이전 Phase 내용을 수정하지 않음. 수정 시 별도 블록으로 추가
- **기록하지 않는 것**: 코드, API, 파일 경로 등 기술 세부사항
- **기록 실패 시**: 상태 전이에 영향 없음. "로그 기록 중 문제 발생. 진행에는 영향 없습니다."

### 이벤트 연결

대화 로그 → 이벤트 단방향 참조만 허용. 역방향(이벤트 → 로그) 금지.

## 이벤트 기록

### Round 완료 시

```typescript
appendScopeEvent(paths, {
  type: "exploration.round_completed",
  actor: "agent",
  payload: {
    phase: 2,
    phase_name: "영역 탐색",
    round: 3,
    topic: "관련 영역 확정",
    decisions: [
      { round: 3, question: "빠진 영역이 있습니까?", answer: "푸시 알림은 제외" },
    ],
    assumptions_found: ["AI 리포트를 홈에 노출 가능"],
    alternatives_presented: ["리포트 요약 + 결제", "리포트만", "전체 리포트"],
    brief_fields_contributed: ["대상 사용자"],
  },
});
```

### Phase 전환 시

```typescript
appendScopeEvent(paths, {
  type: "exploration.phase_transitioned",
  actor: "agent",
  payload: {
    from_phase: 1,
    to_phase: 2,
    reason: "PO가 결과 목록에 동의",
    log_path: "build/exploration-log.md",
    log_hash: contentHash(logContent),
  },
});
```

## 제어 규칙

### 범주 통제

| 문제 | 감지 | 대응 |
|------|------|------|
| 과최적화: 하나의 시나리오에 질문 5개 초과 | 시나리오당 질문 수 | "세부 사항은 Surface에서 구체화. 다음 시나리오로 이동." |
| 과발산: Phase 1의 결과와 무관한 방향 | 결과 목록과 대조 | "원래 결과와 다른 영역. 별도 scope로 분리하시겠습니까?" |
| Phase 역행 | Phase 번호 | "목적이 변경되었습니다. Phase 1부터 다시 정리합니다." |
| 정체: 같은 질문 2회 반복 | 반복 감지 | "보류하고 다음으로 이동. 나중에 돌아올 수 있습니다." |

### Round 상한

- `EXPLORATION_SUMMARY_THRESHOLD` (5회): 5 round 이후 "지금까지의 결정을 정리합니다" 중간 요약 제시
- `MAX_EXPLORATION_ROUNDS` (20회): gate-guard가 `exploration.round_completed`를 거부. "범위를 확정하거나 scope를 보류하세요."

### 완성 판단

Phase 6에서 PO가 범위에 동의하면 완성. 에이전트가 "충분하다"를 일방적으로 판단하지 않음.
단, Phase 6 진입 전에 다음을 확인:
- Phase 1의 모든 결과가 Phase 4 시나리오로 커버되었는가?
- Phase 5의 모든 가정이 확인/수정되었는가?
- 미충족이면 "아직 확인되지 않은 항목이 있습니다"로 안내

### 결정 번복

PO가 이전 결정을 번복하면:
1. 해당 Phase로 돌아감 (backward 이동, 이유 명시)
2. 번복 사유를 exploration-log.md에 기록
3. 의존하는 후속 결정이 있으면 PO에게 알림

### 세션 중단/재개

재개 시 에이전트는:
1. `scope.md` 읽기 → 현재 상태 확인
2. `build/exploration-log.md` 읽기 → 마지막 Phase, 마지막 Round, 합의 내용
3. PO에게 안내:
   > "Phase {N} 진행 중입니다. 시나리오 {M}/{K}까지 완료했습니다.
   > 마지막 대화에서 '{요약}' 방향으로 확정했습니다.
   > 이어서 진행하겠습니까?"

---

## 완료 후

exploration이 완료되면:
1. `exploration-log.md`에서 AlignPacketContent 각 필드를 조립
2. `renderAlignPacket()` 호출하여 Align Packet 생성
3. `align.proposed` 이벤트 기록 (exploration_progress.completed_at 자동 설정)
4. brief.md가 없었다면, Phase 1~6 결과로 brief.md 자동 생성
5. PO에게 Align Packet 제시 → `/align`으로 결정 요청

## 질문 설계 금기

| 금기 | 예시 | 이유 |
|------|------|------|
| 유도 질문 | "프로그레스바를 5단계로 하면 좋지 않을까요?" | PO의 선택이 아닌 sprint-kit의 선택 |
| 기능 쇼핑 | "프로그레스바를 원하시나요? 애니메이션은요?" | 해결책 먼저, 문제 나중 |
