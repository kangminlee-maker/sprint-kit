> 목적: 6개 Phase 전체를 시뮬레이션하여 대화 설계의 누락/모순을 발견

## Phase 1: 목적 정밀화

### sprint-kit의 분석

이 brief는 "상세 brief" 진입입니다. 변경 목적, 대상 사용자, 기대 결과가 모두 채워져 있고, 기대 결과에 구체적 기능(프로그레스바, 플로팅 버튼, 상태별 카드)이 포함되어 있습니다.

brief에서 문제(problem)와 해결책(solution)을 분리합니다:

| 구분 | 내용 |
|------|------|
| 문제 | "학생이 앱에 진입해도 무엇을 해야 하는지 즉시 파악하기 어렵다" |
| 문제 | "체험 완료 후 결제 전환까지의 경로가 끊긴다" |
| 문제 | "예외 상황 발생 시 복귀 경로가 없다" |
| 해결책 | "홈 화면을 상태에 따라 동적으로 전환되는 구조로 재설계" |
| 해결책 | "메인 문구, 프로그레스바, 본문 카드, 플로팅 버튼이 자동 전환" |
| 해결책 | "유료 체험(5,000원/3회)" |

### 시뮬레이션

```
sprint-kit:
brief를 확인했습니다. 구체적인 기능(프로그레스바, 상태별 플로팅 버튼, 본문 카드 전환,
유료 체험 도입)을 구상하셨습니다. 이 기능들이 달성하려는 결과를 먼저 확인하겠습니다.

[결과 1] 학생이 앱에 진입하면 현재 자신의 단계와 다음 행동을 즉시 인지할 수 있다.
[결과 2] 체험을 완료한 학생이 결제까지 도달하는 경로가 끊기지 않는다.
[결과 3] 예외 상황(노쇼, 취소, 소진)에서도 결제 유도 경로에 복귀할 수 있다.

이 결과가 맞습니까?
결과를 확인한 뒤, 그 결과를 가장 잘 달성하는 방법을 함께 탐색하겠습니다.
(구상하신 기능이 최적일 수도 있고, 더 효과적인 방법이 있을 수도 있습니다.)

PO: 맞아. 결과 2가 가장 중요해. 체험은 잘 되는데 결제 전환이 안 돼.

sprint-kit:
확인했습니다. 결과의 우선순위를 반영합니다:
  1순위: 체험 완료 → 결제 전환 경로 확보 (결과 2)
  2순위: 각 단계에서 다음 행동 즉시 인지 (결과 1)
  3순위: 예외 상황에서도 복귀 경로 보장 (결과 3)

기능(프로그레스바, 플로팅 버튼 등)은 Phase 4에서 시나리오별로 최적의 방법을 함께 탐색하겠습니다.
→ Phase 2로 이동
```

### 관찰/발견

- Phase 1에서 PO가 우선순위를 직접 설정함. 이것은 이후 Phase 2(영역 탐색)와 Phase 4(시나리오 탐색)의 순서에 영향을 줌.
- brief의 "유료 체험(5,000원/3회)"은 해결책이자 정책 변경. Phase 5(가정 검증)에서 정책 검증 대상.

## Phase 2: 영역 탐색

### sprint-kit의 분석

목표 "체험→결제 전환 경로 확보"에 대해, 퍼널 기반 탐색이 가장 적합합니다. 체험 시작부터 결제까지의 퍼널을 정의하고, 각 단계의 관련 영역을 식별합니다.

### 시뮬레이션

```
sprint-kit:
목표 "체험→결제 전환 경로 확보"에 관련된 영역을 분석하겠습니다.
체험 시작부터 결제까지의 퍼널을 기준으로 분석합니다.

[퍼널 단계별 관련 영역]

  [1] 체험 신청 → 체험 완료
    관련 영역: 체험 신청 플로우, 레슨 진행, 예외 처리(노쇼/취소)
    현재 상태: 체험 플로우 자체는 동작 중

  [2] 체험 완료 → 앱 재진입
    관련 영역: 홈 화면, 푸시 알림
    현재 상태: 체험 완료 후 별도 상태 없음. 미신청 학생과 동일 화면 표시

  [3] 앱 재진입 → 결제 의사 형성
    관련 영역: 홈 화면(결제 유도), 체험 결과/피드백
    현재 상태: 결제 유도 요소 없음. 체험 결과 요약 페이지 없음

  [4] 결제 의사 → 결제 완료
    관련 영역: 결제 플로우, 수강권 선택 페이지, 가격/혜택 정보
    현재 상태: 결제 플로우 자체는 존재. 체험 후 연결만 없음

[관련 영역 요약]

| 영역 | 경로 위치 | 현재 격차 |
|------|----------|----------|
| 홈 화면 (체험 완료 후) | 필수 경유지 | 격차 큼 — 해당 상태 자체가 없음 |
| 결제 플로우 | 필수 경유지 | 격차 작음 — 플로우 존재, 체험 후 연결만 없음 |
| 체험 결과/피드백 | 선택 경유지 | 격차 큼 — 기능 자체가 없음 |
| 예외 처리 (노쇼/취소/소진) | 선택 경유지 | 격차 큼 — 복귀 경로 없음 |
| 푸시 알림 | 선택 경유지 | 확인 필요 |

제가 확인하지 못한 영역이 있을 수 있습니다.
위 목록에 빠진 영역이 있습니까?

PO: 푸시 알림은 이번에는 빼자. 그리고 체험 결과/피드백은... 현재 레슨 완료 후
    AI가 생성하는 학습 리포트가 있긴 한데, 홈 화면에는 노출이 안 돼.

sprint-kit:
확인했습니다.
  - 푸시 알림: 이번 scope에서 제외합니다.
  - AI 학습 리포트: 기존 기능이 존재하지만 홈 화면에 연결되지 않은 상태.
    이것은 "체험 결과/피드백" 영역에서 활용할 수 있는 기존 자산입니다.

영역 목록을 확정합니다:
  1. 홈 화면 (체험 완료 후 상태) — 핵심
  2. 결제 플로우 (체험 후 연결) — 핵심
  3. 체험 결과/피드백 (AI 학습 리포트 홈 노출) — 핵심
  4. 예외 처리 (노쇼/취소/소진 복귀 경로) — 연관

이 영역들에 대해 현재 상태를 상세히 확인하겠습니다.
→ Phase 3으로 이동
```

### 관찰/발견

- **PO가 sprint-kit이 모르는 정보를 추가함**: "AI 학습 리포트가 있다"는 사실은 brownfield 스캔에서 발견될 수도 있고, PO만 아는 정보일 수도 있음. 이 시뮬레이션에서는 PO가 직접 알려줌.
- **PO가 영역을 제외함**: 푸시 알림을 이번 scope에서 명시적으로 제외. 이것은 Phase 6(범위 합의)에서 "제외 범위"에 기록됨.
- **설계 문서에 누락된 부분 발견**: Phase 2에서 "PO가 기존 자산을 알려줄 때"의 처리 규칙이 없음. "AI 학습 리포트"처럼 sprint-kit이 스캔에서 발견했을 수도 있는 기능을 PO가 먼저 언급하는 경우, sprint-kit은 brownfield에서 해당 기능의 현재 구현 상태를 확인하여 보고해야 함.

## Phase 3: 현재 상태 공유

### 시뮬레이션

```
sprint-kit:
Phase 2에서 확정된 4개 영역의 현재 동작을 설명드리겠습니다.

[영역 1: 홈 화면]
현재 홈 화면은 학생의 상태에 따라 4가지로 분기됩니다:
  1. 수강권이 없는 학생 → "수업 추천" 배너 + 예약 버튼
  2. 레슨이 예약된 학생 → 다음 레슨 정보 카드 + 입장 버튼
  3. 체험 수강권이 있는 학생 → 체험 수업 예약 유도
  4. 정규 수강권이 있는 학생 → 정규 수업 예약 유도

체험을 완료한 학생은 1번("수강권이 없는 학생")으로 분류되어,
체험 전 학생과 동일한 화면을 봅니다.

[영역 2: 결제 플로우]
GNB의 "수강신청" 버튼을 통해 수강권 선택 → 결제 페이지에 접근할 수 있습니다.
체험 완료 후 이 버튼 이외에 결제로 이어지는 경로는 없습니다.

[영역 3: 체험 결과/피드백]
레슨 완료 후 AI 학습 리포트가 생성됩니다.
현재 이 리포트는 레슨 상세 페이지에서만 접근 가능하고,
홈 화면에서는 노출되지 않습니다.

[영역 4: 예외 처리]
노쇼, 취소, 소진 상태는 현재 별도 분기가 없습니다.
- 튜터 노쇼: 시스템이 체험 1회를 자동 복구하지만, 학생에게 알림이 가지 않음
- 학생 노쇼: 잔여 횟수가 차감되지만, 홈 화면에 잔여 횟수 안내 없음
- 전체 소진: "수강권이 없는 학생"과 동일하게 처리됨

이 이해가 맞습니까?

PO: 맞아. 튜터 노쇼 시 자동 복구 되는 건 맞는데, 학생 입장에서는
    "왜 레슨이 취소됐지?" 상태로 방치되는 거지. 알림은 카카오 알림톡으로 가긴 하는데
    앱 안에서는 아무것도 안 보여.

sprint-kit:
확인했습니다. 추가 정보를 반영합니다:
  - 튜터 노쇼 시: 체험 1회 자동 복구 + 카카오 알림톡 발송 (앱 내 안내 없음)
  - 학생 입장에서 앱 내에서 상황을 인지할 수 없는 것이 핵심 격차입니다.

→ Phase 4로 이동
```

### 관찰/발견

- **PO가 앱 외부 채널(카카오 알림톡) 정보를 추가함**: brownfield 스캔으로는 발견하기 어려운 정보. 외부 시스템 연동 정보는 PO에게 직접 확인해야 할 수 있음.
- **설계 문서에 누락된 부분 발견**: Phase 3에서 "앱 외부 채널(알림톡, 이메일, SMS 등)의 현재 동작"을 확인하는 절차가 없음. 영역 탐색(Phase 2)에서 앱 외부 채널을 터치포인트로 언급했지만, Phase 3에서 실제로 확인하는 질문이 빠져 있음.

## Phase 4: 시나리오 탐색

### 시나리오 순서 결정

Phase 1의 우선순위(결과 2 > 결과 1 > 결과 3)와 Phase 2의 영역 목록을 교차하여 시나리오 순서를 결정합니다:

1. **체험 완료 후 첫 앱 진입** (결과 2 — 최우선)
2. **체험 완료 후 재진입** (결과 2 — 반복 방문)
3. **미신청 학생 첫 진입** (결과 1)
4. **대기 중 학생** (결과 1)
5. **레슨 중/완료 직후** (결과 1)
6. **노쇼/취소 후 진입** (결과 3)
7. **전체 소진 후 진입** (결과 3)

### 시나리오 1: 체험 완료 후 첫 앱 진입 (최우선)

```
sprint-kit:
가장 중요한 시나리오부터 탐색하겠습니다.

[상황] 체험 레슨을 정상 완료한 학생이 다음 날 앱을 열었습니다.

질문 1: 이 학생이 홈 화면에서 가장 먼저 인지해야 하는 것은 무엇입니까?
질문 2: 이 학생이 취해야 할 다음 행동은 무엇입니까?
질문 3: 이 학생이 결제하지 않고 이탈한다면, 가장 가능성 높은 이유는 무엇입니까?

PO: 체험이 끝났다는 걸 명확히 보여주고, 바로 결제 혜택을 보여줘야 해.
    최대한 빨리 결제 페이지로 가게 해야 해.
    이탈 이유는... 가격 부담, 체험 불만족, 지금 당장 결정 필요 못 느낌.

sprint-kit:
확인했습니다. "상태 확인 → 결제 혜택 → 결제 페이지" 방향입니다.

참고로, 체험 완료 학생을 결제로 전환하는 접근은 이것 외에도 있습니다:

  (A) 현재 방향 — 즉시 결제 유도
      체험 완료 확인 + 혜택 전면 노출 + 결제 CTA.

  (B) 가치 환기 후 유도
      Phase 3에서 확인한 AI 학습 리포트를 홈 화면에 먼저 노출하여
      "체험에서 이만큼 배웠다"는 가치를 환기한 뒤 결제로 유도.

  (C) 다음 체험 예약으로 자연스러운 전환
      잔여 체험이 남아있으면 다음 예약을 유도하면서,
      예약 과정에서 정규 수강권 혜택을 비교 노출.

현재 방향(A)을 유지하시겠습니까?

PO: A로 가되, B의 요소도 섞자. AI 학습 리포트 요약을 짧게 보여주고
    바로 결제 혜택으로 연결하면 좋겠어.

sprint-kit:
확정합니다. "AI 학습 리포트 요약(짧게) → 결제 혜택 노출 → 결제 CTA" 순서입니다.
```

### 시나리오 2: 체험 완료 후 재진입

```
sprint-kit:
[상황] 체험 완료 후 결제를 하지 않은 학생이 3일 뒤에 다시 앱을 열었습니다.

질문 1: 이 학생에게 첫 진입과 동일한 화면을 보여줍니까, 다른 화면을 보여줍니까?
질문 2: 결제 혜택의 강도를 유지합니까, 높이거나 낮춥니까?

PO: 같은 화면이면 안 돼. 이미 한 번 봤으니까 "또 이거야" 하고 무시할 거야.
    혜택을 좀 더 구체적으로 보여주거나, 기한을 붙여야 할 것 같아.

sprint-kit:
확인했습니다. 체험 완료 후 "첫 진입"과 "재진입"이 다른 화면이어야 합니다.
이것은 상태 분기에 "체험 완료 - 첫 방문"과 "체험 완료 - 재방문"의 구분이 필요하다는 의미입니다.

재방문 화면의 방향:
  (A) 혜택 기한 강조 — "첫 수강 30% 할인, N일 남음"
  (B) 소셜 증거 — "이번 주 체험 후 결제한 학생 N명"
  (C) 재체험 유도 — "한 번 더 체험해 보시겠습니까?"

어느 방향이 적합합니까?

PO: A. 기한을 붙이면 행동 유도가 될 거야.

sprint-kit:
확정합니다. 재방문 시 "혜택 기한 강조" 방향입니다.
```

### Phase 4 전체 관찰/발견

- **7개 시나리오를 통해 상태 분기가 구체화됨**: 미신청, 대기, 레슨 직후, 체험 완료(첫 방문), 체험 완료(재방문), 노쇼/취소, 전체 소진. brief의 "6~7가지"보다 세분화됨.
- **brief에 없던 구분이 자연스럽게 도출됨**: "레슨 직후"와 "체험 완료 후 재진입"이 다른 화면이어야 한다는 것은 brief에 명시되지 않았으나, 시나리오 탐색에서 PO가 직접 "다를 수 있어"라고 판단.
- **"체험 완료 후 재방문" 시나리오에서 "혜택 기한" 아이디어가 도출됨**: brief에 없던 요소. Phase 4의 대안 제시에서 나온 것이 아니라 PO 자신의 발상. 이것도 추출→제안→확정 패턴에서 "추출" 단계에서 나옴.
- **설계 문서에 누락된 부분 발견**: 시나리오 탐색 순서를 "Phase 1 우선순위 × Phase 2 영역"의 교차로 결정하는 규칙이 설계 문서에 명시되지 않음.

### 관찰/발견

- **Phase 5에서 scope가 축소됨**: "유료 체험"이 제외됨. brief에 명시되어 있던 기능이 가정 검증을 통해 제외됨. 이것이 Adaptive Align의 핵심 가치 — brief의 기능이 무조건 구현되는 것이 아니라, 검증을 거쳐 범위가 조정됨.
- **가정의 유형별 처리가 다름**: 정책 가정(A4, A5)은 온톨로지/약관으로 즉시 확인. 기술 전제(A6, A7)는 "구현 단계에서 확인"으로 보류. 사용자 행동 가정(A1, A2)은 PO 판단에 의존.
- **설계 문서에 누락된 부분 발견**: "기술 전제 가정"의 처리 방법이 명확하지 않음. "구현 단계에서 확인"이라고 했는데, 만약 구현 단계에서 기술적으로 불가능한 것으로 판명되면 어떻게 되는가? 이것은 compile 또는 apply 단계에서 `constraint_gap_found`로 역전이해야 하는 상황인데, 현재 프로토콜에서 이 경로가 명시적으로 연결되지 않음.

- **brief 대비 변경 사항**:
