# Adaptive Align — 대화 설계 (Conversation Design)

> 상태: 설계 초안 (draft)

> 선행 문서: adaptive-align-proposal.md
> 근거: 인터뷰 방법론 (Contextual Inquiry, JTBD, Assumption Mapping, Story Mapping)

---

## 1. 설계 원칙

### 구현이 아니라 대화를 먼저 설계한다

이벤트 스키마, 상태 머신, 타입 정의는 대화가 성공적으로 이루어진 후에 필요한 인프라입니다.
먼저 답해야 할 질문: **"어떤 순서로 무엇을 물어야, PO의 모호한 의도가 실행 가능한 명세가 되는가?"**

### 인터뷰 방법론에서 차용하는 4가지 원칙

**Contextual Inquiry**: 현재 상태를 sprint-kit이 먼저 제시하고, PO가 확인/수정합니다. PO에게 현재 상태를 설명하라고 요구하지 않습니다.

**Jobs-to-be-Done (JTBD)**: 기능이 아니라 시나리오를 탐색합니다. "프로그레스바를 몇 단계로 할지"가 아니라 "학생이 앱을 열었을 때 어떤 경험을 하길 원하는지"를 묻습니다.

**Assumption Mapping**: 모든 결정에 깔린 암묵적 가정을 명시적으로 드러냅니다. PO가 인지하지 못한 가정을 sprint-kit이 발견하여 확인을 요청합니다.

**Story Mapping**: 전체 흐름을 먼저 넓게 그리고, 각 부분을 점진적으로 깊이 들어갑니다.

### 질문 설계의 금기


| 금기                 | 예시                                   | 이유                            |
| ------------------ | ------------------------------------ | ----------------------------- |
| 유도 질문              | "프로그레스바를 5단계로 하면 좋지 않을까요?"           | PO의 선택이 아닌 sprint-kit의 선택을 제시 |
| 기능 쇼핑              | "프로그레스바를 원하시나요? 애니메이션은요?"            | 해결책을 먼저 제시하고 문제를 뒤에 붙이는 구조    |
| 맥락 없는 질문           | "상태를 몇 개로 하시겠습니까?"                   | PO가 현재 상태를 모르면 판단할 수 없음       |
| 이진 질문 (개방형이 필요할 때) | "노쇼 시 자동 소진하시겠습니까?"                  | 가능한 선택지가 2개뿐인 것처럼 프레이밍        |
| 기술 프레이밍            | "useGreetingStatus 훅의 분기를 어떻게 바꿀까요?" | PO는 개발자가 아님                   |

## 2. 6개 Phase 구조

### 전체 흐름

```
Phase 1: 목적 정밀화     — "달성하려는 결과는 무엇인가?"
Phase 2: 영역 탐색       — "이 결과에 어떤 영역이 관련되는가?"
Phase 3: 현재 상태 공유   — "각 영역이 지금 어떻게 동작하는가?"
Phase 4: 시나리오 탐색    — "각 영역에서 사용자가 어떤 경험을 하길 원하는가?"
Phase 5: 가정 검증       — "이 방향에 어떤 가정이 깔려 있는가?"
Phase 6: 범위 합의       — "이번에 바꾸는 것과 바꾸지 않는 것"
```

### Phase 간 관계

Phase는 순차적이지만 엄격하지 않습니다. Phase 4에서 Phase 1의 목적이 수정될 수 있고, Phase 2에서 발견한 영역이 Phase 4의 시나리오를 추가할 수 있습니다. 다만 **역방향 이동 시에는 이유를 명시**합니다: "시나리오를 탐색하면서 원래 목적을 수정할 필요가 생겼습니다."

## 3. 진입 방식: brief는 입력이 아니라 산출물

### brief의 역할 전환

현재 `/start` 프로토콜은 brief.md를 필수 입력으로 요구합니다. PO가 "변경 목적", "대상 사용자", "기대 결과"를 직접 작성해야 grounding이 시작됩니다.

Adaptive Align에서 brief는 **6개 Phase의 산출물**입니다. PO는 brief를 작성할 필요가 없습니다. 대화를 통해 결정된 내용이 자동으로 brief에 해당하는 정보를 구성합니다.

### 3가지 진입 방식


| 진입 방식        | PO가 제공하는 것                | sprint-kit의 대응                                            |
| ------------ | ------------------------- | --------------------------------------------------------- |
| **대화만**      | 구두 한 문장 ("체험 후 결제가 안 돼")  | Phase 1부터 시작. 6개 Phase가 brief 전체를 구성                      |
| **간략 brief** | 변경 목적만 작성, 나머지 비어 있음      | Phase 1에서 목적을 확인한 후, 비어 있는 부분을 Phase 2~6에서 채움             |
| **상세 brief** | 모든 필드가 채워져 있음 (homeui 사례) | Phase 1에서 목적을 재정위하고, brief에 이미 결정된 부분은 Phase에서 확인만 하고 건너뜀 |


### 각 Phase가 brief의 어떤 부분을 생성하는가


| brief 필드  | 생성하는 Phase                          | 비고                                   |
| --------- | ----------------------------------- | ------------------------------------ |
| 변경 목적     | Phase 1 (목적 정밀화)                    | PO의 시작점을 결과(outcome) 수준으로 재정위한 결과    |
| 대상 사용자    | Phase 2 (영역 탐색) + Phase 4 (시나리오 탐색) | 영역에서 영향받는 사용자 그룹 + 시나리오의 행위자         |
| 기대 결과     | Phase 1 (목적 정밀화) + Phase 6 (범위 합의)  | Phase 1의 결과 목록 + Phase 6에서 확정된 포함 범위 |
| 포함 범위     | Phase 6 (범위 합의)                     | 6개 Phase의 종합 결과                      |
| 제외 범위     | Phase 6 (범위 합의)                     | Phase 2에서 발견되었으나 이번에 제외하기로 한 영역      |
| 제약 및 참고사항 | Phase 5 (가정 검증)                     | 확인된 가정, 미확인 가정, 정책 제약                |


### `/start` 프로토콜과의 관계

현재 `/start`의 Path A는 brief 템플릿을 생성하고, Path B는 brief를 검증한 후 grounding을 시작합니다. Adaptive Align에서는:

- **brief 없이 시작**: PO가 `/start {projectName}`을 실행하면 scope 폴더 생성 후 즉시 대화 시작. brief.md는 Phase 6 완료 후 자동 생성.
- **brief 있이 시작**: 기존 Path B와 동일하게 brief 검증 후 grounding 시작. 단, brief의 빈 필드가 있으면 해당 Phase에서 채움.
- **grounding 시점**: brief 없이 시작한 경우, Phase 2(영역 탐색)에서 "어떤 소스를 스캔해야 하는가"가 결정되므로, grounding은 Phase 2 이후에 실행됩니다. 상세 brief가 있는 경우는 기존대로 즉시 grounding 가능.

## 4. 각 Phase의 상세 설계

### Phase 1: 목적 정밀화

**목표**: PO가 어떤 수준에서 대화를 시작하든, sprint-kit이 그것을 **"달성하려는 결과(outcome)"** 수준으로 재정위(reframe)합니다. 모든 후속 Phase는 이 결과를 기준으로 진행됩니다.

**방법**: PO의 시작점을 파악하고, 그 시작점에서 목표를 역추적합니다.

#### PO의 시작점 유형과 재정위 방법

PO는 다양한 수준에서 대화를 시작할 수 있습니다. sprint-kit은 어떤 시작점이든 **"어떤 결과를 달성하려 하는가?"**로 대화를 재정위합니다.


| 시작점 유형                  | PO 예시                                  | sprint-kit의 재정위                                                                                          |
| ----------------------- | -------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **목표(outcome)**         | "수업 예약 수를 늘리고 싶어"                      | 이미 목표 수준. "수업 예약 수가 현재 어떤 상태이고, 어느 지점에서 이탈이 발생하는지 확인해 보겠습니다."                                            |
| **해결책(solution)**       | "홈화면에서 바로 예약 가능한 가장 빠른 시간을 보여주자"       | "예약 가능한 시간을 먼저 보여주면 달성하려는 결과는 무엇입니까? 예약 진입 허들을 낮추는 것인지, 빈 시간대 매칭률을 높이는 것인지에 따라 방향이 달라집니다."               |
| **문제(pain point)**      | "학생들이 체험 수업 끝나고 그냥 이탈해"                | "체험 완료 후 이탈이 핵심 문제입니다. 이 이탈을 줄이면 궁극적으로 달성하려는 결과는 무엇입니까? 결제 전환율 향상인지, 재방문율 향상인지, 다른 것인지 확인하겠습니다."         |
| **기능 명세(feature spec)** | "프로그레스바를 추가하고, 플로팅 버튼 라벨을 상태별로 바꾸고..." | "구체적인 기능을 구상하셨군요. 이 기능들이 최종적으로 달성하려는 결과는 무엇입니까? 기능은 결과를 달성하기 위한 수단이므로, 결과를 먼저 확인하면 더 적합한 수단이 있을 수 있습니다." |
| **지표(metric)**          | "체험→결제 전환율을 15%에서 25%로 올리고 싶어"         | "전환율 목표를 확인했습니다. 현재 15%인 전환율에서 이탈이 발생하는 지점이 어디인지 파악하면, 10%p를 어디서 확보할 수 있는지 알 수 있습니다."                    |


#### 재정위의 원칙

**1. PO의 시작점을 부정하지 않습니다.**
"그것은 해결책이지 목표가 아닙니다"라고 말하지 않습니다. PO가 제시한 것을 인정한 뒤, "그것이 달성하려는 결과"를 묻습니다.

**2. 결과(outcome)는 "달라지는 것"으로 정의합니다.**
"프로그레스바를 추가한다"는 결과가 아닙니다. "학생이 자신의 현재 단계를 즉시 인지할 수 있다"가 결과입니다. 결과는 항상 **사용자 또는 비즈니스의 상태 변화**로 표현됩니다.

**3. 결과가 확인되면 brief의 해결책은 보류합니다.**
PO가 제시한 기능(프로그레스바, 플로팅 버튼 등)은 Phase 4의 시나리오 탐색에서 대안과 함께 재검토합니다. Phase 1에서는 "그 기능으로 달성하려는 결과"만 확인합니다.

#### 질문 형식 — 진입 방식별

sprint-kit의 첫 응답은 PO의 진입 방식에 따라 달라집니다.

**대화만으로 시작한 경우** (brief 없음):

```
PO: 체험 후 결제가 안 돼.

sprint-kit: 확인했습니다. 말씀하신 내용에서 핵심 결과(달성 목표)를 정리하겠습니다.

[결과 1] 체험을 완료한 학생이 결제까지 도달하는 경로가 끊기지 않는다.

이것이 이번에 달성하려는 핵심 결과가 맞습니까?
추가로 달성하려는 결과가 있으면 말씀해 주세요.
```

**간략 brief가 있는 경우** (변경 목적만 작성):

```
sprint-kit: brief의 변경 목적을 확인했습니다. 핵심 결과를 정리하겠습니다.

[결과 1] 학생이 앱에 진입하면 "다음에 무엇을 해야 하는지" 즉시 파악할 수 있다.
[결과 2] 체험을 완료한 학생이 결제까지 도달하는 경로가 끊기지 않는다.

이 결과가 맞습니까?
혹시 빠진 결과가 있거나, 우선순위가 다르다면 말씀해 주세요.
```

**상세 brief가 있는 경우** (기능 명세까지 포함 — homeui 사례):

```
sprint-kit: brief를 확인했습니다. 구체적인 기능(프로그레스바, 상태별 플로팅 버튼, 본문 카드 전환)을
구상하셨습니다. 이 기능들이 달성하려는 결과를 먼저 확인하겠습니다.

[결과 1] 학생이 앱에 진입하면 현재 자신의 단계와 다음 행동을 즉시 인지할 수 있다.
[결과 2] 체험 완료 후 결제까지의 경로가 끊기지 않는다.
[결과 3] 예외 상황에서도 재진입 경로가 보장된다.

이 결과가 맞습니까?
결과를 확인한 뒤, 그 결과를 가장 잘 달성하는 방법을 함께 탐색하겠습니다.
(구상하신 기능이 최적일 수도 있고, 더 효과적인 방법이 있을 수도 있습니다.)
```

**이 Phase에서 sprint-kit이 하는 일**:

- PO의 시작점(목표/해결책/문제/기능/지표)을 파악
- 시작점이 목표 수준이 아니면 "그것이 달성하려는 결과"를 역추적하여 재정위
- brief가 있으면 문제(problem)와 해결책(solution)을 분리. brief가 없으면 PO의 발화에서 추출
- 해결책은 인정하되 보류: "Phase 4에서 대안과 함께 검토합니다"
- PO가 추가한 결과가 있으면 목록에 반영

**이 Phase에서 sprint-kit이 하지 않는 일**:

- PO의 시작점을 부정하거나 교정하지 않음 ("그것은 목표가 아닙니다" → 금지)
- 해결 방향을 제안하지 않음 (Phase 4에서 시나리오를 통해 도출)
- 기존 시스템의 기술적 제약을 언급하지 않음 (Phase 2에서 별도로 다룸)
- 범위를 확정하지 않음 (Phase 5에서 최종 확인)

**전환 기준**: PO가 결과(outcome) 목록에 동의하면 Phase 2로 이동.

### Phase 2: 영역 탐색

**목표**: Phase 1에서 확인된 목표(결과)에 **어떤 서비스 영역이 관련되는가**를 체계적으로 발견합니다. sprint-kit이 brownfield와 온톨로지를 기반으로 관련 영역을 제시하고, PO가 확인/추가합니다.

**왜 이 Phase가 필요한가**: 목표가 "체험→결제 전환율 향상"이면, 관련 영역은 홈 화면만이 아닙니다. 결제 플로우, 알림 시스템, 체험 후 리텐션 메커니즘, 가격 정책 페이지 등이 모두 관련될 수 있습니다. 어떤 영역이 관련되는지를 먼저 파악해야, Phase 3에서 "어떤 시나리오를 탐색할 것인가"가 결정됩니다. 영역 파악 없이 시나리오를 탐색하면, 홈 화면만 집중적으로 개선하고 결제 플로우의 이탈은 방치하는 결과가 나올 수 있습니다.

#### 영역 탐색의 4가지 방법

sprint-kit은 목표의 성격에 따라 적합한 탐색 방법을 선택합니다. 하나의 방법만 사용할 수도 있고, 여러 방법을 조합할 수도 있습니다.

**방법 1: 엔티티 기반 Top-down 탐색**

목표에 관련된 도메인 엔티티(온톨로지에 정의된 것)를 찾고, 그 엔티티가 관여하는 모든 화면/기능/정책을 추적합니다.

```
목표: "체험→결제 전환율 향상"
  ↓ 관련 엔티티 식별
  체험 수강권(TrialTicket) → 이 엔티티가 관여하는 영역:
    - 체험 신청 플로우
    - 체험 레슨 진행
    - 체험 완료 후 홈 화면
    - 수강권 만료/소진 처리
  정규 수강권(RegularTicket) → 이 엔티티가 관여하는 영역:
    - 결제 플로우 (수강권 선택 → 결제)
    - 가격/혜택 정보 페이지
    - 수강권 비교 화면
  레슨(Lesson) → 이 엔티티가 관여하는 영역:
    - 레슨 예약
    - 레슨 진행 (입장/완료)
    - 레슨 결과 (피드백, AI 학습 결과)
```

적합한 경우: 목표가 특정 도메인 개념(수강권, 레슨)과 직접 연결되는 경우.

**방법 2: 퍼널(funnel) 기반 탐색**

목표 달성까지의 사용자 여정을 단계별로 나누고, 각 단계에서 이탈이 발생할 수 있는 영역을 식별합니다.

```
목표: "체험→결제 전환율 향상"
  ↓ 전환 퍼널 정의
  [1단계] 체험 완료 → 앱 재진입
    관련 영역: 푸시 알림, 홈 화면 진입점
  [2단계] 앱 재진입 → 결제 의사 형성
    관련 영역: 홈 화면(결제 유도), 체험 결과 요약, 혜택 안내
  [3단계] 결제 의사 → 결제 페이지 도달
    관련 영역: CTA 버튼, 결제 페이지 진입 경로
  [4단계] 결제 페이지 → 결제 완료
    관련 영역: 가격 정책, 결제 UI, 할인/프로모션
```

적합한 경우: 목표가 전환율, 완료율 같은 퍼널 지표와 연결되는 경우.

**방법 3: 터치포인트 매핑**

목표에 관련된 사용자와 서비스의 접점을 모두 열거하고, 각 접점이 목표에 기여하는지/방해하는지를 평가합니다.

```
목표: "체험→결제 전환율 향상"
  ↓ 터치포인트 열거
  - 홈 화면 (체험 완료 후 진입) → 현재: 결제 유도 없음
  - 푸시 알림 (체험 완료 후 발송) → 현재: 발송 여부 미확인
  - 이메일 (체험 완료 후) → 현재: 없음
  - GNB "수강신청" 버튼 → 현재: 항상 노출
  - 체험 결과 페이지 → 현재: 없음
  - 고객 지원 (CS) → 현재: 결제 문의 대응만
```

적합한 경우: 목표가 사용자 경험 전반에 걸쳐 있고, 단일 퍼널로 설명되지 않는 경우.

**방법 4: 영향 체인(impact chain) 역추적**

목표에서 출발하여 "이 결과가 달성되려면 무엇이 필요한가?"를 재귀적으로 추적합니다.

```
목표: "체험→결제 전환율 향상"
  ↓ 이 결과가 달성되려면?
  "체험 완료 학생이 결제 페이지에 도달해야 한다"
    ↓ 이것이 달성되려면?
    "학생이 결제할 이유를 인지해야 한다" → 관련 영역: 혜택 안내
    "학생이 결제 페이지에 쉽게 접근할 수 있어야 한다" → 관련 영역: CTA, 네비게이션
    ↓ 이것이 달성되려면?
    "학생이 앱에 재진입해야 한다" → 관련 영역: 푸시 알림, 리텐션
    "학생이 체험에 만족했어야 한다" → 관련 영역: 체험 품질 (scope 밖일 수 있음)
```

적합한 경우: 목표가 복합적이고, 어떤 영역이 관련되는지가 직관적이지 않은 경우.

#### sprint-kit의 영역 제시와 PO의 추가

sprint-kit은 위 방법 중 하나 이상을 사용하여 관련 영역 목록을 제시합니다. 동시에, **sprint-kit이 감지하지 못한 영역이 있을 수 있음**을 명시합니다.

```
sprint-kit:
목표 "체험→결제 전환율 향상"에 관련된 영역을 분석했습니다.

[관련 영역 — sprint-kit 분석 결과]
  1. 홈 화면 (체험 완료 후 상태) — 현재 별도 상태 없음
  2. 결제 플로우 (수강권 선택 → 결제 완료) — 현재 존재하나 체험 후 연결 없음
  3. 체험 결과/피드백 — 현재 레슨 완료 후 표시되는 정보 없음
  4. 예외 처리 (노쇼/취소/소진) — 현재 복귀 경로 없음

[제가 확인하지 못한 영역이 있을 수 있습니다]
  위 목록에 빠진 영역이 있습니까?
  예를 들어, 앱 외부 채널(푸시 알림, 이메일, CS 등)이나
  오프라인 접점이 이 전환에 영향을 미치는 경우가 있을 수 있습니다.
```

#### PO가 기존 자산/기능을 알려줄 때의 처리

PO가 "AI 학습 리포트가 있다", "카카오 알림톡으로 리마인드를 보내고 있다"처럼 기존 자산을 언급하면, sprint-kit은 다음을 수행합니다:

1. **brownfield에서 해당 기능의 구현 상태를 확인**: "AI 학습 리포트는 레슨 상세 페이지에서 접근 가능합니다. 홈 화면에는 노출되지 않습니다."
2. **brownfield에서 확인되지 않으면 PO에게 상세를 요청**: "그 기능이 앱의 어디에서 어떻게 동작하는지 알려주시면 확인하겠습니다."

이 정보는 Phase 3(현재 상태 공유)의 입력을 보강하고, Phase 4(시나리오 탐색)에서 기존 자산을 활용하는 대안을 제시하는 데 사용됩니다.

#### PO가 새 영역을 제시했을 때의 검증

PO가 sprint-kit이 제시하지 않은 영역을 추가하면, sprint-kit은 그 영역이 목표와 실제로 관련되는지를 **2가지 기준**으로 검증합니다.

**기준 1: 인과 관계 — 이 영역의 변경이 목표 달성에 영향을 미치는가?**

```
PO: "튜터 프로필 페이지도 관련 있어. 체험 전에 튜터 프로필을 보고 기대감이 생겨야 체험 후 결제까지 가거든."

sprint-kit:
확인합니다. "튜터 프로필 → 기대감 형성 → 체험 만족도 → 결제 전환"의 인과 경로입니다.
brownfield를 확인하니, 현재 튜터 프로필 페이지에는 경력, 자기소개, 평점이 표시됩니다.
이 영역을 이번 scope에 포함할 경우, 변경 대상은 "튜터 프로필에 체험 관련 정보 추가"가 됩니다.
관련 영역 목록에 추가하시겠습니까?
```

**기준 2: 영향 범위 — 이 영역의 변경이 목표 외의 다른 기능에 영향을 미치는가?**

```
PO: "결제 가격 자체를 바꾸는 것도 생각하고 있어."

sprint-kit:
가격 변경은 체험→결제 전환율에 직접적으로 영향을 미칩니다.
다만, 가격 변경은 기존 정규 수강생의 결제 조건에도 영향을 미칩니다.
이것은 이번 scope("체험 후 전환 경로")보다 넓은 범위입니다.
이번 scope에 포함하시겠습니까, 아니면 별도 scope로 분리하시겠습니까?
```

#### PO가 sprint-kit이 제시하지 않은 영역을 언급할 때

```
PO: "튜터 매칭 알고리즘도 바꿔야 하지 않을까?"

sprint-kit:
확인했습니다. 매칭 알고리즘이 "체험→결제 전환"에 미치는 영향을 확인하겠습니다.

현재 분석 결과, 매칭 알고리즘은 체험 "신청 시점"에 작동하고,
체험 "완료 후" 전환 경로에는 직접적인 인과 관계가 파악되지 않았습니다.

다만, 제가 파악하지 못한 연결이 있을 수 있습니다.
매칭 알고리즘이 전환에 영향을 준다고 판단하신 근거가 있으시면 알려주세요.
그 근거를 바탕으로 이번 scope에 포함할지 함께 판단하겠습니다.
```

**원칙**: sprint-kit도 PO도 항상 100% 완벽할 수 없습니다.

- sprint-kit이 인과 관계를 발견하지 못했다고 해서 "관련 없습니다"라고 단정하지 않습니다
- 동시에, 불확실한 연결을 "영향을 줄 수 있습니다"라고 추측하여 제시하지도 않습니다
- "직접적인 인과 관계가 파악되지 않았다"는 사실을 전달하고, PO가 가진 맥락을 요청합니다
- PO가 근거를 제시하면 함께 검토하고, 근거가 없으면 이번 scope에서 제외합니다

#### 영역이 많을 때의 점진적 탐색

관련 영역이 5개를 넘으면, 모든 영역을 동시에 다루기 어렵습니다. 이 경우 **목표 기여도 순으로 정렬**하여 점진적으로 탐색합니다.

```
sprint-kit:
관련 영역이 7개 발견되었습니다. 목표 기여도가 높은 순서로 정렬합니다:

[핵심 영역 — 목표에 직접 영향]
  1. 홈 화면 (체험 완료 후 상태) — 전환 경로의 진입점
  2. 결제 플로우 — 전환 경로의 종착점
  3. 체험 결과/피드백 — 결제 동기 형성

[연관 영역 — 목표에 간접 영향]
  4. 예외 처리 (노쇼/취소/소진) — 전환 경로 이탈 후 복귀
  5. 푸시 알림 — 앱 재진입 유도
  6. 튜터 프로필 — 체험 기대감 형성
  7. 가격 정책 — 결제 허들

먼저 핵심 영역(1~3)에 대해 Phase 3(시나리오 탐색)을 진행하고,
연관 영역(4~7)은 이번 scope에 포함할지 Phase 6(범위 합의)에서 결정하시겠습니까?
아니면 연관 영역부터 포함 여부를 먼저 결정하시겠습니까?
```

**전환 기준**: PO가 영역 목록에 동의하고, Phase 3에서 탐색할 핵심 영역이 확정되면 Phase 3로 이동.

### Phase 3: 현재 상태 공유

**목표**: Phase 2에서 확정된 각 영역에 대해, sprint-kit이 brownfield 분석 결과를 PO가 이해할 수 있는 언어로 설명하여 공유 기반을 만듭니다.

**방법**: Contextual Inquiry 방식. sprint-kit이 관찰 결과를 먼저 제시하고, PO가 확인/수정합니다.

**질문 형식**:

```
sprint-kit: 소스를 분석한 결과, 현재 홈 화면의 동작을 설명드리겠습니다.

[현재 동작]
홈 화면은 학생의 상태에 따라 4가지로 분기됩니다:
  1. 수강권이 없는 학생 → "수업 추천" 배너 + 예약 버튼
  2. 레슨이 예약된 학생 → 다음 레슨 정보 카드 + 입장 버튼
  3. 체험 수강권이 있는 학생 → 체험 수업 예약 유도
  4. 정규 수강권이 있는 학생 → 정규 수업 예약 유도

[현재 없는 것]
  - "체험 완료 후 결제 유도" 상태가 없습니다
  - "노쇼/취소/소진" 예외 상태가 없습니다
  - 프로그레스바(단계 표시기)가 없습니다

이 이해가 맞습니까? 실제 서비스에서 다르게 동작하는 부분이 있으면 알려주세요.
```

**brownfield 정보의 번역 규칙**:


| 코드 분석 결과                          | PO에게 보여주는 형식                           |
| --------------------------------- | -------------------------------------- |
| `greetingStatusSchema`에 4개 enum 값 | "홈 화면은 4가지 상태로 분기됩니다: ..."             |
| `useGreetingStatus` 훅의 조건 분기      | "학생의 상태를 판단하는 기준은 수강권 유무, 레슨 예약 여부입니다" |
| `HomePageView` 컴포넌트 구조            | "홈 화면에는 배너, 인사말, 수업 준비 카드가 있습니다"       |
| API 엔드포인트 목록                      | 보여주지 않음 (PO에게 불필요)                     |
| DB 스키마                            | 보여주지 않음                                |
| 모듈 의존성                            | 보여주지 않음                                |


**원칙**: PO에게 보여주는 것은 "사용자가 보는 것"과 "사용자가 할 수 있는 것"입니다. "시스템이 어떻게 구현되어 있는가"는 보여주지 않습니다. 단, PO가 질문하면 설명합니다.

**앱 외부 채널 확인 (표준 질문)**:

현재 상태 설명 후, 반드시 다음 질문을 합니다:

```
sprint-kit:
위는 앱 안에서 학생이 보는 것입니다.
앱 밖에서 학생에게 전달되는 것이 있습니까?
(예: 카카오 알림톡, 문자, 이메일, 전화 등)
```

brownfield 스캔은 앱 코드만 분석하므로, 앱 외부 채널은 PO에게 직접 확인해야 합니다. 이 정보는 Phase 4의 시나리오 탐색에서 "학생이 앱에 재진입하는 경로"를 이해하는 데 필수적입니다.

**이 Phase에서 sprint-kit이 하는 일**:

- brownfield 스캔 결과를 "사용자 경험" 언어로 번역
- 현재 동작과 brief가 요구하는 것 사이의 차이(gap)를 명시
- 앱 외부 채널의 현재 동작을 PO에게 확인
- PO의 수정/보충 요청을 반영

**이 Phase에서 sprint-kit이 하지 않는 일**:

- 변경 방향을 제안하지 않음 (Phase 4에서 시나리오를 통해 도출)
- 기술적 난이도를 언급하지 않음
- "이것은 어렵습니다" 또는 "이것은 쉽습니다" 같은 판단을 하지 않음

**전환 기준**: PO가 현재 상태 설명(앱 내부 + 외부 채널 모두)에 동의하면 Phase 4로 이동.

---

### Phase 3: 시나리오 탐색

**목표**: Phase 1의 문제를 해결하는 구체적인 사용자 시나리오를 PO와 함께 정의합니다. 기능이 아니라 "사용자가 어떤 상황에서 어떤 경험을 하는가"를 탐색합니다.

**방법**: JTBD + Story Mapping + Design Ideation. 핵심 시나리오(happy path)를 먼저 다루고, 예외 시나리오를 후속으로 다룹니다.

**Phase 3의 핵심 원칙: 추출(extract)과 제안(generate)의 균형**

sprint-kit은 PO의 의도를 묻기만 하는 것이 아닙니다. PO가 생각하지 못한 가능성을 적극적으로 제안합니다. PO의 경험은 자신이 사용하고 관찰한 서비스의 범위에 한정됩니다. sprint-kit은 온톨로지(도메인 규칙), brownfield(기존 시스템의 숨겨진 기능), 그리고 유사 서비스의 일반적 패턴을 기반으로 PO가 상상하지 못한 대안을 제시할 수 있습니다.

**대안 제시의 시점과 형태**:

대안은 **PO의 초기 의도를 확인한 후에** 제시합니다. PO가 먼저 자기 생각을 말하기 전에 대안을 제시하면 유도 질문이 됩니다. PO가 답한 후에 "이런 방향도 가능합니다"로 시야를 넓힙니다.

```
[나쁜 순서]
sprint-kit: "이런 방법이 있습니다: A, B, C. 어떤 것이 좋으십니까?"  → PO의 독자적 사고를 차단

[좋은 순서]
PO: "결제 페이지로 바로 보내야 해."
sprint-kit: "직접 결제 유도 방향을 확인했습니다. 참고로, 유사한 상황에서 다른 접근도 있습니다:
  - 결제 전에 체험 요약을 보여주어 만족감을 환기시킨 뒤 결제로 유도하는 방식
  - 결제를 강제하지 않고, 다음 체험 예약을 유도하면서 자연스럽게 유료 전환을 노출하는 방식
  현재 방향(즉시 결제 유도)을 유지하시겠습니까, 아니면 다른 방향을 검토하시겠습니까?"
```

**대안 제시의 3가지 소스**:


| 소스                      | 제시 형태                              | 예시                                                                                               |
| ----------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------ |
| **brownfield (기존 시스템)** | "현재 시스템에 이미 이런 기능이 있는데 활용할 수 있습니다" | "현재 수강신청 페이지에 수강권 비교 기능이 이미 있습니다. 홈 화면에서 바로 결제 페이지가 아닌 비교 페이지로 유도하는 것도 가능합니다."                   |
| **온톨로지 (도메인 규칙)**       | "이 도메인에서 일반적으로 사용되는 패턴이 있습니다"      | "교육 서비스에서 체험 후 전환율은 '완료 직후'보다 '2일 내 재방문' 시점이 높은 경우가 많습니다. 홈 화면에 '지난 체험 돌아보기'를 먼저 보여주는 것도 방법입니다." |
| **설계 패턴 (UX 일반)**       | "이런 상황에서 효과적인 접근이 있습니다"            | "이탈 방지에서 '긴급성 부여'(혜택 기한 설정)와 '가치 재확인'(체험 결과 요약) 중 어느 쪽이 이 서비스에 적합할지 고려해 보시겠습니까?"                 |


**대안 제시의 금기**:


| 금기                              | 이유                                       |
| ------------------------------- | ---------------------------------------- |
| PO가 답하기 전에 대안 먼저 제시             | PO의 독자적 사고를 차단. 유도 질문이 됨                 |
| "이것이 더 좋습니다"라는 판단 포함            | sprint-kit은 옵션을 제시하지 판정하지 않음. PO가 판단     |
| 대안을 5개 이상 나열                    | 선택 과부하. 기존 방향 + 대안 1~2개가 적절              |
| 기술 구현 난이도를 대안 선택 기준으로 제시        | PO는 "무엇이 좋은가"로 판단해야지 "무엇이 쉬운가"로 판단하면 안 됨 |
| PO가 이미 확신을 가진 방향을 반복적으로 재고하게 만듦 | 1회 대안 제시 후 PO가 원래 방향을 유지하면 존중            |


**시나리오 탐색 순서 결정 규칙**:

시나리오 순서는 다음 3단계 기준으로 결정합니다:

1. **1차 정렬 — Phase 1 결과 우선순위**: 결과 A 관련 시나리오를 먼저, 결과 B 관련 시나리오를 나중에
2. **2차 정렬 — Phase 2 격차 크기**: 격차가 큰 영역의 시나리오를 먼저
3. **3차 정렬 — 사용자 빈도**: 빈번한 시나리오(happy path)를 먼저, 예외 시나리오를 나중에

예: homeui scope에서 결과 A(결제 전환)가 1순위이고, 홈 화면의 격차가 가장 크므로, "체험 완료 후 홈 화면 재진입"이 첫 번째 시나리오가 됩니다.

**질문 형식 — 시나리오 1개당 (추출 → 제안 → 확정)**:

```
[Step 1 — 추출: PO의 의도를 먼저 확인]

sprint-kit: 첫 번째 시나리오를 탐색하겠습니다.

[상황] 무료체험을 아직 신청하지 않은 학생이 앱을 처음 열었습니다.

질문 1: 이 학생이 홈 화면에서 가장 먼저 인지해야 하는 것은 무엇입니까?
질문 2: 이 학생이 "다음 행동"으로 취해야 할 것은 무엇입니까?
질문 3: 이 학생이 홈 화면에서 이탈하지 않으려면, 어떤 정보가 부족해서는 안 됩니까?

PO: (답변)

[Step 2 — 제안: PO의 답변을 확인한 후, 대안을 제시]

sprint-kit: 확인했습니다. {PO의 의도 요약}.
참고로, 이 시나리오에서 다른 접근도 가능합니다:
  - {대안 A: brownfield 기반}
  - {대안 B: 온톨로지/패턴 기반}
현재 방향을 유지하시겠습니까, 아니면 다른 방향을 검토하시겠습니까?

PO: (선택)

[Step 3 — 확정: 최종 방향 기록]

sprint-kit: {최종 선택} 방향으로 확정합니다.
```

**시나리오별 질문 패턴**:


| 시나리오 유형      | 질문 초점                               | 대안 제시 초점                           |
| ------------ | ----------------------------------- | ---------------------------------- |
| 첫 진입 (미신청)   | "무엇을 인지 → 무엇을 행동 → 무엇이 부족하면 이탈"     | 온보딩 패턴의 대안 (단계형 vs 즉시 행동 vs 탐색 허용) |
| 대기 중 (예약 완료) | "레슨 전까지 무엇을 하길 원하는가 → 입장 시점에 어떤 변화" | 대기 시간 활용 방안 (사전 학습 vs 커뮤니티 vs 대기만) |
| 레슨 완료        | "완료 직후 감정 → 다음 행동 → 결제 유도 시점"       | 전환 유도 패턴 (즉시 vs 지연 vs 복합)          |
| 예외 (노쇼)      | "노쇼 인지 → 복구 행동 → 책임 귀속"             | 복구 경험 패턴 (보상형 vs 재시도형 vs 안내형)      |


**이 Phase에서 sprint-kit이 하는 일**:

- 시나리오를 하나씩 순서대로 제시
- 각 시나리오에서 PO의 의도를 먼저 확인 (추출)
- PO의 답변 확인 후, brownfield/온톨로지/설계 패턴 기반 대안을 제시 (제안)
- PO의 최종 선택을 기록 (확정)
- PO의 답변에서 암묵적 가정을 발견하면 기록 (Phase 4에서 검증)

**이 Phase에서 sprint-kit이 하지 않는 일**:

- PO가 답하기 전에 대안을 먼저 제시하지 않음
- 대안을 "더 좋은 것"으로 프레이밍하지 않음 (옵션으로만 제시)
- PO가 유지를 선택한 방향을 다시 재고하게 만들지 않음
- 기술적 제약을 이유로 시나리오를 제한하지 않음
- 한 시나리오에서 너무 깊이 들어가지 않음 (시나리오당 질문 3~5개 제한)

**전환 기준**: 핵심 시나리오(brief에서 언급된 사용자 상태 수)를 모두 다루면 Phase 4로 이동. homeui의 경우 6가지 상태가 brief에 명시되어 있으므로 6개 시나리오를 탐색합니다.

### Phase 4: 가정 검증

**목표**: Phase 1~3에서 축적된 결정에 깔린 암묵적 가정을 명시적으로 드러내고, PO에게 확인을 요청합니다.

**방법**: Assumption Mapping. 각 결정에서 "이것이 사실이어야 이 결정이 유효하다"는 전제를 추출합니다.

**가정의 3가지 유형**:


| 유형        | 예시                                | 위험도                   |
| --------- | --------------------------------- | --------------------- |
| 사용자 행동 가정 | "체험 완료 직후 결제 의향이 가장 높다"           | 높음 — 틀리면 결제 유도 시점이 무효 |
| 서비스 정책 가정 | "튜터 노쇼 시 체험 1회가 자동 복구된다"          | 높음 — 정책 변경이 필요할 수 있음  |
| 기술 전제 가정  | "현재 API로 학생의 체험 상태를 실시간 판별할 수 있다" | 중간 — 구현 범위에 영향        |


**질문 형식**:

```
sprint-kit: 지금까지의 논의를 정리하면서, 몇 가지 가정을 발견했습니다. 확인이 필요합니다.

[가정 1 — 사용자 행동]
"체험 레슨을 완료한 학생은 곧바로 결제할 의향이 있다."
→ 이 가정이 맞다면: 체험 완료 직후 홈 화면에서 결제 CTA를 전면 노출합니다.
→ 이 가정이 틀리다면: 체험 후 일정 기간(예: 1~2일) 후에 결제 유도가 더 효과적일 수 있습니다.
이 가정이 맞습니까?

[가정 2 — 서비스 정책]
"무료체험 3회를 모두 소진한 학생에게 유료 체험(5,000원/3회)을 제안한다."
→ 이것은 현재 운영 중인 정책입니까, 아니면 이번에 새로 도입하려는 정책입니까?
→ 현재 약관(이용약관 제14조)에 유료 체험 관련 조항이 있습니까?

[가정 3 — 서비스 정책]
"튜터 노쇼 시 체험 1회가 자동 복구된다."
→ 현재 이 정책이 시스템에 구현되어 있습니까?
→ 학생 노쇼와 튜터 노쇼의 처리가 다릅니까?
```

**이 Phase에서 sprint-kit이 하는 일**:

- Phase 1~3의 결정에서 암묵적 가정을 추출
- 각 가정에 대해 "맞다면 어떻게 되고, 틀리다면 어떻게 되는가"를 제시
- 정책 가정은 온톨로지/이용약관 소스와 교차 대조하여 "확인됨/미확인" 표시
- PO의 확인 결과에 따라 이전 Phase의 시나리오를 수정할 수 있음

**이 Phase에서 sprint-kit이 하지 않는 일**:

- 가정이 틀렸다고 판정하지 않음 (PO가 판단)
- 기술적 실현 가능성을 이유로 가정을 거부하지 않음

**모순 발견 처리**: 이 Phase에서 가정 간 모순이 발견됩니다.

- 예: "노쇼 시 자동 복구" + "3회 소진 시 유료 전환" → 노쇼가 무한 복구되면 3회 소진에 도달하지 않음
- 모순 발견 시 즉시 PO에게 알리고, 두 가정 중 하나를 수정하도록 요청

**기술 전제 가정의 보류 규칙**:

기술 전제("이것이 기술적으로 가능한가")를 "구현 단계에서 확인"으로 보류할 때, 반드시 **불가능 시 대안**을 함께 기록합니다:

```
[가정 — 기술 전제]
"AI 학습 리포트를 홈에 요약 노출할 수 있다"
→ 가능하면: 시나리오 1, 6 그대로 진행
→ 불가능하면: 리포트 요약 대신 "체험을 완료했습니다" 상태 메시지 + 결제 경로만 표시
→ 판단 권한: 대안이 Phase 1의 결과(outcome)를 여전히 달성하면 Builder가 판단.
             대안이 결과 자체를 변경하면 PO 확인 필요.
```

이렇게 하면 compile/apply 단계에서 기술적 불가능이 발견되었을 때, `constraint_gap_found`로 역전이하지 않고 대안으로 진행할 수 있는 경우와, PO 확인이 필요한 경우가 구분됩니다.

**scope 변경 시 시나리오 역수정 규칙**:

가정 검증의 결과로 항목이 scope에서 제외되거나 가정이 기각되면:

1. 해당 가정/항목에 의존하는 Phase 4 시나리오를 식별
2. 영향받는 시나리오의 변경 사항을 PO에게 즉시 안내
3. PO가 변경된 시나리오를 승인하면 Phase 6으로 진행
4. PO가 변경을 거부하면 해당 시나리오를 Phase 4에서 재탐색

예: "유료 체험"이 제외되면 → 시나리오 6(전체 소진)이 영향받음 → "유료 체험 없이 정규 결제만 안내"로 변경 → PO에게 확인

**전환 기준**: 추출된 가정 전부에 대해 PO가 확인/수정을 완료하고, scope 변경으로 영향받는 시나리오의 수정도 승인되면 Phase 6으로 이동.

### Phase 5: 범위 합의

**목표**: Phase 1~4의 결과를 종합하여, 이번 scope에서 "바꾸는 것"과 "바꾸지 않는 것"을 명시적으로 확정합니다. 이것이 Align Packet의 `proposed_scope.in/out`이 됩니다.

**방법**: Story Mapping의 "Release 1 경계 긋기". 전체 시나리오 중 이번에 포함할 것과 제외할 것을 PO와 합의합니다.

**질문 형식**:

```
sprint-kit: Phase 1~4의 결과를 정리했습니다. 이번 scope의 범위를 확정하겠습니다.

[포함 — 이번에 바꾸는 것]
  ✓ 홈 화면 상태 분기를 4가지에서 7가지로 확장
    (미신청, 대기, 레슨 중, 완료, 튜터 노쇼, 학생 노쇼/취소, 전체 소진)
  ✓ 각 상태에서의 메인 문구, CTA 버튼 라벨/동작 자동 전환
  ✓ 체험 완료 후 결제 유도 화면 (혜택 카드 + CTA)
  ✓ 예외 상태에서의 복귀 경로 (재신청 유도)

[제외 — 이번에 바꾸지 않는 것]
  ✗ 프로그레스바 (단계 표시기) — Phase 3에서 필수가 아닌 것으로 확인됨
  ✗ 백엔드 API 변경 — 기존 API로 상태 판별 가능
  ✗ 유료 체험(5,000원/3회) 결제 플로우 — 약관 검토 후 별도 scope
  ✗ GNB "수강신청" 버튼 변경 — 현재 동작 유지

[확인된 가정]
  • 체험 완료 직후 결제 유도가 적절하다 (PO 확인)
  • 튜터 노쇼 시 체험 1회 자동 복구 (현재 정책과 일치, 온톨로지에서 확인)
  • 학생 노쇼 시 잔여 횟수 차감 (PO 확인, 약관 제14조와 일치)

[미확인 가정 — 이번 scope 진행 중 확인 필요]
  • 유료 체험 정책의 약관 반영 여부 (법무 검토 필요)

이 범위에 동의하십니까?
```

**이 Phase에서 sprint-kit이 하는 일**:

- Phase 1~4의 결과를 `포함/제외/확인된 가정/미확인 가정` 4개 섹션으로 정리
- 각 포함 항목이 어떤 시나리오(Phase 3)에서 도출되었는지 추적
- 제외 항목에 제외 이유를 명시 (Phase 3에서 불필요 판정, 별도 scope 분리, 기존 유지 등)

**전환 기준**: PO가 범위에 동의하면, 이 내용이 Align Packet의 입력이 됩니다.

## 5. Phase 간 제어 규칙

### 범주 통제 (과최적화/과발산 방지)


| 문제                                  | 감지 기준                     | 대응                                                                   |
| ----------------------------------- | ------------------------- | -------------------------------------------------------------------- |
| 과최적화: 하나의 시나리오에 질문이 5개를 초과          | Phase 3에서 시나리오당 질문 수 모니터링 | "이 시나리오의 세부 사항은 Surface 단계에서 구체화됩니다. 다음 시나리오로 이동합니다."                |
| 과발산: PO가 brief에 없는 새 기능을 요구         | Phase 1의 문제 목록과 대조        | "이 내용은 원래 문제(X)와 다른 영역입니다. 별도 scope로 분리하시겠습니까, 아니면 이번 범위에 포함하시겠습니까?" |
| Phase 역행: Phase 4에서 Phase 1의 문제를 변경 | Phase 번호 모니터링             | "목적 자체가 변경되었습니다. Phase 1부터 다시 정리하겠습니다." (역행 사유 명시)                   |
| 정체: 동일 시나리오에서 PO가 결정을 미루는 경우        | 같은 질문이 2회 이상 반복           | "이 시나리오는 보류하고 다음으로 이동합니다. 나중에 돌아올 수 있습니다."                           |


### 완성 판단

Phase 5에서 PO가 범위에 동의하면 완성입니다. 에이전트가 "충분하다"를 일방적으로 판단하지 않습니다. 단, 에이전트는 Phase 5 진입 전에 다음을 확인합니다:

- Phase 1의 모든 문제가 Phase 3의 시나리오로 커버되었는가?
- Phase 4의 모든 가정이 확인/수정되었는가?
- 하나라도 미충족이면 Phase 5 진입 전에 "아직 확인되지 않은 항목이 있습니다"로 안내

### 결정 번복 처리

PO가 이전 결정을 번복하면:

- 해당 Phase로 돌아감
- 번복 사유를 기록
- 번복된 결정에 의존하는 후속 결정이 있으면 PO에게 알림: "이 결정을 변경하면, Phase 3에서 결정한 X도 재검토가 필요합니다"

### 세션 중단/재개

중단 시 현재 Phase와 진행 상태를 기록합니다:

```
"Phase 3 진행 중 (시나리오 3/6 완료).
완료된 시나리오: 미신청, 대기, 레슨 중.
남은 시나리오: 완료, 노쇼, 소진."
```

## 6. 대화 전문 기록 (Exploration Log)

### 왜 대화 전문을 기록하는가

이벤트 로그(`events.ndjson`)는 **무엇이 결정되었는가**(what)를 구조화된 데이터로 기록합니다. 그러나 **왜 그렇게 결정했는가**(why)의 맥락 — sprint-kit이 어떤 brownfield 정보를 제시했고, 어떤 대안을 보여줬고, PO가 왜 그것을 선택했는지 — 는 이벤트에 담기지 않습니다.

이 맥락은 두 가지 상황에서 필요합니다:
- **세션 재개**: 중단된 세션을 이어갈 때, 이전 대화의 흐름을 복원해야 합니다
- **장기 추적**: 3개월 후 "CST-003을 왜 defer로 결정했는가?"를 확인할 때, `rationale` 한 줄이 아닌 대화 맥락이 필요합니다

### 파일과 형식

**경로**: `scopes/{scope-id}/build/exploration-log.md`

**이유**: `build/` 디렉토리에는 이미 "이벤트에서 재생성 불가능한 산출물"(align-packet.md, build-spec.md, prd.md)이 저장되는 패턴이 확립되어 있습니다. 대화 전문도 동일한 범주입니다.

**형식**: 구조화된 마크다운. 소비자가 LLM 에이전트(세션 재개 시 읽기)와 PO(결정 근거 확인) 두 종류이므로, 둘 다 읽을 수 있는 형식이어야 합니다.

```markdown
# Exploration Log — {scope-id}

## Phase 1: 목적 정밀화
- **상태**: 완료
- **확정된 결과**:
  1. 체험 완료 학생 → 결제 전환 경로 확보 (1순위)
  2. 각 단계에서 다음 행동 즉시 인지 (2순위)
  3. 예외 상황에서도 복귀 경로 보장 (3순위)

### Round 1
**sprint-kit**: brief에서 3가지 결과를 추출했습니다:
[결과 1] 학생이 앱 진입 시 다음 행동을 즉시 인지할 수 있다.
[결과 2] 체험 완료 후 결제 경로가 끊기지 않는다.
[결과 3] 예외 상황에서도 복귀 경로가 보장된다.
이 결과가 맞습니까?

**PO**: 맞아. 결과 2가 가장 중요해.

**결정**: 결과 우선순위 확정 — 2 > 1 > 3

---

## Phase 2: 영역 탐색
- **상태**: 완료
- **확정된 영역**: 홈 화면, 결제 플로우, 체험 결과/피드백, 예외 처리
- **제외**: 푸시 알림 (PO 판단)

### Round 1
**sprint-kit**: 퍼널 기반으로 4개 영역을 분석했습니다:
| 영역 | 경로 위치 | 현재 격차 |
|------|----------|----------|
| 홈 화면 | 필수 | 격차 큼 |
| 결제 플로우 | 필수 | 격차 작음 |
| 체험 결과/피드백 | 선택 | 격차 큼 |
| 예외 처리 | 선택 | 격차 큼 |
빠진 영역이 있습니까?

**PO**: 푸시 알림은 빼자. AI 학습 리포트가 있긴 한데 홈에는 안 보여.

**sprint-kit**: AI 학습 리포트는 레슨 상세 페이지에서 접근 가능합니다. 홈에 노출되지 않습니다.

**결정**: 영역 4개 확정. 푸시 알림 제외. AI 학습 리포트 기존 자산 확인.

---

## Phase 4: 시나리오 탐색
- **상태**: 진행 중 (시나리오 3/7)
...
```

### 기록 규칙

**1. 기록 시점**: 매 round 종료 시 에이전트가 `exploration-log.md`에 append합니다.

**2. append-only**: 이전 Phase/Round의 내용을 수정하지 않습니다. Phase 5(가정 검증)에서 이전 시나리오가 수정되면, 원본은 유지하고 수정 사항을 별도 블록으로 추가합니다:
```markdown
### 수정 — Phase 5 가정 검증에서 반영
시나리오 7(전체 소진) 변경: "유료 체험 또는 결제" → "정규 결제만"
사유: 유료 체험이 약관 미반영으로 scope에서 제외됨
```

**3. 기록 실패 시**: 대화 로그 기록은 관찰적 활동입니다. 실패해도 상태 전이에 영향 없음. PO에게 "대화 로그 기록 중 문제가 발생했습니다. 진행에는 영향 없습니다."로 안내합니다.

**4. 기록하지 않는 것**: 코드 스니펫, API 경로, 파일 경로 등 기술 세부사항. 그것은 events.ndjson과 state/ 파일의 역할입니다. 대화 로그는 "왜"만 담습니다.

### 이벤트 로그와의 관계

대화 로그는 이벤트 로그를 **보완**합니다. 중복도 대체도 아닙니다.

| 기록 대상 | events.ndjson | exploration-log.md |
|-----------|---------------|-------------------|
| "Phase 2에서 4개 영역 확정" | 이벤트 payload에 구조화 기록 | 불필요 (이벤트로 충분) |
| "PO가 푸시 알림을 제외한 이유" | 기록할 곳 없음 | **여기** |
| "sprint-kit이 A+B 복합 대안을 제시하고 PO가 선택한 과정" | 기록할 곳 없음 | **여기** |
| CST-003의 결정이 inject | constraint.decision_recorded | 불필요 (이벤트로 충분) |

**참조 방향**: 대화 로그 → 이벤트(단방향만 허용). 대화 로그의 각 Round에 관련 이벤트의 `revision` 번호를 포함할 수 있습니다. 역방향(이벤트가 대화 로그를 참조)은 금지합니다. reducer가 대화 로그에 의존하면 GC-013(결정론)이 깨집니다.

### 이벤트 연결

GC-007(산출물 생성 시 이벤트 기록)에 따라, Phase 전환 시점에 이벤트를 기록합니다:

```typescript
appendScopeEvent(paths, {
  type: "exploration.phase_transitioned",
  actor: "agent",
  payload: {
    from_phase: 1,
    to_phase: 2,
    reason: "PO가 결과 목록에 동의",
    log_path: "build/exploration-log.md",
    log_hash: contentHash(logContent),
  },
});
```

### 세션 재개 시 활용

세션 재개 시 에이전트는 두 파일을 읽습니다:
1. `scope.md` → 구조적 위치 (현재 상태, 미결정 건수)
2. `build/exploration-log.md` → 의미적 맥락 (마지막 Phase, 마지막 Round, 합의 내용)

재개 안내 예시:
```
"Phase 4(시나리오 탐색) 진행 중입니다.
시나리오 3/7까지 완료했습니다 (미신청, 대기, 레슨 직후).
마지막 대화에서 '레슨 직후 화면은 축하 → AI 리포트 → 결제 유도' 방향으로 확정했습니다.
다음 시나리오는 '체험 완료 후 첫 앱 진입'입니다."
```

### scope.md에 Phase 요약 표시

대화 전문은 `exploration-log.md`에 저장되지만, PO가 매번 전문을 읽지 않아도 되도록 `scope.md`에 Phase 요약을 표시합니다:
- 현재 Phase: 펼쳐서 표시 (합의 내용 + 다음 행동)
- 이전 Phase: `<details>` 태그로 접힘

이 요약은 `exploration-log.md`의 각 Phase 헤더 아래 "확정된 결과/영역/시나리오" 목록에서 추출됩니다.

### downstream 참조

```
exploration-log.md ──→ Align Packet 생성 (에이전트가 Phase 6 결과를 AlignPacketContent로 변환)
                   ──→ brief.md 자동 생성 (Phase 6 완료 시)
                   ──→ PRD의 "Pre-Apply Review" 참조 (Phase 5 가정 검증 결과)
```

Align Packet 생성 시 에이전트는 exploration-log.md를 읽고 `AlignPacketContent`의 각 필드를 채웁니다. 이 변환은 코드가 아니라 에이전트 프로토콜에서 수행합니다 ("탐지는 코드, 판정은 에이전트" 원칙).

## 7. homeui scope 시뮬레이션

### Phase 1 시뮬레이션

**시뮬레이션 A — homeui brief (해결책에서 시작하는 경우)**

homeui의 brief는 목표와 해결책이 혼합되어 있습니다. "메인 문구, 프로그레스바, 본문 카드, 플로팅 버튼이 자동 전환"은 해결책이고, "각 단계에서 다음 행동을 유도"는 목표입니다.

```
sprint-kit:
brief를 읽었습니다. 구체적인 기능(프로그레스바, 상태별 플로팅 버튼, 본문 카드 전환)을
구상하셨습니다. 이 기능들이 달성하려는 결과를 먼저 확인하겠습니다.

제가 이해한 핵심 결과는 다음과 같습니다:
  [결과 1] 학생이 앱에 진입하면 현재 자신의 단계와 다음 행동을 즉시 인지할 수 있다.
  [결과 2] 체험을 완료한 학생이 결제까지 도달하는 경로가 끊기지 않는다.
  [결과 3] 예외 상황(노쇼, 취소, 소진) 발생 시에도 결제 유도 경로에 복귀할 수 있다.

이 결과가 맞습니까?
결과를 확인한 뒤, 그 결과를 가장 잘 달성하는 방법을 함께 탐색하겠습니다.
(구상하신 기능이 최적일 수도 있고, 더 효과적인 방법이 있을 수도 있습니다.)

PO: 맞아. 그리고 결과 2가 가장 중요해. 체험은 잘 되는데 결제 전환이 안 되는 게 핵심이야.

sprint-kit:
확인했습니다. 결과의 우선순위를 반영합니다:
  1순위: 체험 완료 → 결제 전환 경로 확보 (결과 2)
  2순위: 각 단계에서 다음 행동 즉시 인지 (결과 1)
  3순위: 예외 상황에서도 복귀 경로 보장 (결과 3)

기능(프로그레스바, 플로팅 버튼 등)은 Phase 3에서 시나리오별로 최적의 방법을 함께 탐색하겠습니다.
→ Phase 2로 이동
```

**시뮬레이션 B — 만약 PO가 "수업 예약 수를 늘리고 싶어"로 시작했다면 (목표에서 시작)**

```
sprint-kit:
목표를 확인했습니다. "수업 예약 수 증가"입니다.
이 목표를 더 구체적으로 이해하기 위해 확인이 필요합니다.

  질문 1: 예약 수가 부족한 구간이 어디입니까?
    - 체험 수업의 예약 수인지, 정규 수업의 예약 수인지
    - 신규 학생의 첫 예약인지, 기존 학생의 재예약인지

  질문 2: 예약까지의 경로에서 이탈이 발생하는 지점이 어디라고 판단하십니까?
    - 앱 진입 후 예약 페이지에 도달하지 못하는 것인지
    - 예약 페이지에 도달하지만 예약을 완료하지 않는 것인지

PO: 체험 수업 예약은 잘 되는데, 체험 끝나고 정규 수업으로 전환이 안 돼.

sprint-kit:
확인했습니다. "수업 예약 수 증가"의 구체적 의미는
"체험 완료 학생의 정규 수업 전환(결제)"입니다.

핵심 결과를 이렇게 정리하겠습니다:
  [결과 1] 체험을 완료한 학생이 정규 수업 결제까지 도달하는 경로가 끊기지 않는다.

이것이 맞습니까? 추가로 달성하려는 결과가 있으면 말씀해 주세요.
```

**관찰**: 시작점이 "수업 예약 수"라는 지표 수준이었지만, sprint-kit의 재정위 질문을 통해 "체험→정규 전환"이라는 구체적 결과로 수렴했습니다. brief에 "프로그레스바"나 "플로팅 버튼" 같은 해결책이 처음부터 없으므로, Phase 3에서 더 열린 시나리오 탐색이 가능합니다.

### Phase 2 시뮬레이션

```
sprint-kit:
소스를 분석한 결과, 현재 홈 화면의 동작을 설명드리겠습니다.

[현재 동작]
홈 화면은 학생의 상태에 따라 4가지로 분기됩니다:
  1. 수강권이 없는 학생 → "수업 추천" 배너 + 예약 버튼
  2. 레슨이 예약된 학생 → 다음 레슨 정보 카드 + 입장 버튼
  3. 체험 수강권이 있는 학생 → 체험 수업 예약 유도
  4. 정규 수강권이 있는 학생 → 정규 수업 예약 유도

[문제 2와의 관계]
현재 상태 분기에 "체험 완료 후" 상태가 없습니다.
체험을 완료한 학생은 "수강권이 없는 학생"(1번)으로 분류되어,
체험 전 학생과 동일한 화면을 봅니다.
즉, "체험을 완료했다"는 경험이 홈 화면에 전혀 반영되지 않습니다.

[문제 3과의 관계]
노쇼, 취소, 소진 상태도 기존 4가지 분기에 포함되지 않습니다.
노쇼가 발생한 학생은 "체험 수강권이 있는 학생"(3번)으로 분류됩니다.
노쇼 사실을 인지시키거나 대응 행동을 유도하는 화면이 없습니다.

이 이해가 맞습니까?

PO: 맞아. 체험 완료한 학생이 미신청 학생과 같은 화면을 보는 게 가장 큰 문제야.

sprint-kit:
확인했습니다. 체험 완료 학생이 별도 상태로 분류되지 않는 것이 핵심 격차입니다.
→ Phase 3으로 이동
```

### Phase 3 시뮬레이션 (핵심 시나리오 1개 — 추출→제안→확정 전체 흐름)

```
sprint-kit:
가장 중요한 시나리오부터 탐색하겠습니다. (우선순위 1: 결제 전환)

[시나리오: 체험 완료 후 첫 앱 진입]
체험 레슨을 정상 완료한 학생이 다음 날 앱을 열었습니다.

── Step 1: 추출 ──

질문 1: 이 학생이 홈 화면에서 가장 먼저 인지해야 하는 것은 무엇입니까?
질문 2: 이 학생이 홈 화면에서 취해야 할 다음 행동은 무엇입니까?

PO: 체험이 끝났다는 걸 명확히 보여주고, 바로 결제 혜택을 보여줘야 해.
     최대한 빨리 결제 페이지로 가게 해야 해.

── Step 2: 제안 ──

sprint-kit:
확인했습니다. "상태 확인 → 결제 혜택 노출 → 결제 페이지 이동" 방향입니다.

참고로, 체험 완료 학생을 결제로 전환하는 접근은 이것 외에도 있습니다:

  (A) 현재 방향 — 즉시 결제 유도
      체험 완료 상태 확인 + 혜택 전면 노출 + 결제 CTA.
      장점: 전환 경로가 짧음. 단점: 결정 부담이 높아 이탈 가능.

  (B) 가치 환기 후 유도
      체험에서 학습한 내용 요약(AI 분석 결과, 튜터 피드백)을 먼저 보여준 뒤
      "이 학습을 이어가려면"이라는 맥락에서 결제로 유도.
      장점: 체험의 가치를 재확인시켜 결제 동기 부여. 단점: 전환 경로가 1단계 길어짐.

  (C) 다음 체험 예약으로 자연스러운 전환
      체험이 아직 남아있으면 다음 체험 예약을 유도하면서,
      예약 과정에서 정규 수강권의 혜택을 비교 노출.
      장점: 부담 없는 재참여 → 자연스러운 전환. 단점: 체험 3회를 모두 써야 결제 접점.

현재 방향(A)을 유지하시겠습니까, 아니면 다른 방향을 검토하시겠습니까?

PO: A로 가되, B의 요소도 섞자. 체험 결과 요약을 짧게 보여주고 바로 결제 혜택으로 연결하면 좋겠어.

── Step 3: 확정 ──

sprint-kit:
확정합니다. "체험 결과 요약(짧게) → 결제 혜택 노출 → 결제 CTA" 순서입니다.
A와 B의 복합 방향으로 기록합니다.

질문 3: 이 학생이 결제를 하지 않고 이탈한다면, 가장 가능성 높은 이유는 무엇이라고
생각하십니까?

PO: 가격이 부담되거나, 체험이 만족스럽지 않았거나, 지금 당장 결정할 필요를 못 느끼거나.

sprint-kit:
3가지 이탈 이유를 기록합니다. 특히 3번("지금 당장 결정할 필요를 못 느낌")은
Phase 4에서 "체험 직후가 최적 시점인가" 가정 검증에 사용됩니다.

→ 다음 시나리오로 이동
```

**관찰**: PO의 최초 답변은 "즉시 결제 유도(A)"였습니다. sprint-kit이 대안(B, C)을 제시하자 PO는 A+B 복합 방향을 선택했습니다. 대안을 제시하지 않았다면 "즉시 결제 유도"로만 진행되었을 것이고, "체험 결과 요약"이라는 요소가 누락되었을 것입니다. 이것이 **추출만으로는 얻을 수 없고, 제안을 통해서만 도달할 수 있는 결과**입니다.

### Phase 4 시뮬레이션 (핵심 가정 1개만 예시)

```
sprint-kit:
시나리오 탐색에서 발견된 가정을 확인합니다.

[가정 1 — 사용자 행동]
"체험 완료 직후가 결제 전환의 최적 시점이다."

Phase 3에서 PO님이 말씀하신 이탈 이유 중 하나가 "지금 당장 결정할 필요를 못 느낌"이었습니다.
이것은 "체험 직후 = 최적 시점"이라는 가정과 충돌할 수 있습니다.

두 가지 방향이 있습니다:
  (A) 체험 직후에 강하게 유도하되, 이탈 시 다음 진입 때 다시 유도
  (B) 체험 직후에는 가볍게 안내하고, 일정 기간 후 적극 유도

어느 방향이 적합합니까?

PO: A. 체험 직후에 한 번 강하게 보여주고, 그래도 안 하면 다음에 또 보여주자.

sprint-kit:
확인했습니다. "체험 직후 강력 유도 + 이후 진입 시 반복 유도" 방향입니다.
이것은 시나리오 "체험 완료 후 첫 진입"과 "체험 완료 후 재진입"이 다른 화면이 될 수 있음을 의미합니다.
[기록: 상태 분기에 "체험 완료 후 첫 진입"과 "체험 완료 후 재진입" 구분이 필요할 수 있음]
```

## 9. 데이터 기반 강화: ClickHouse 연동

### 개요

`.sprint-kit.yaml`에 ClickHouse MCP가 소스로 등록되어 있으면, sprint-kit은 사용자 행동 이벤트 데이터를 조회하여 각 Phase의 판단을 데이터로 보강합니다. 연결되지 않은 환경에서도 6개 Phase는 코드 분석 + PO 판단만으로 동작합니다.

### Phase별 데이터 활용


| Phase       | ClickHouse 없이                | ClickHouse 연결 시                                 |
| ----------- | ---------------------------- | ----------------------------------------------- |
| 1. 목적 정밀화   | PO의 주관적 문제 서술                | 현재 지표 수치를 목표에 부착: "체험→결제 전환율이 현재 5.8%입니다"       |
| 2. 영역 탐색    | 코드 경로 분석(경로 위치, 격차)만으로 영역 추정 | **퍼널 데이터로 이탈 지점 특정**: "홈→결제 8%, 푸시→재진입 23%"     |
| 3. 현재 상태 공유 | "홈 화면은 4가지 상태로 분기됩니다"        | "4가지 상태 중 상태 1(수강권 없음)에 해당하는 학생이 전체의 61%입니다"    |
| 4. 시나리오 탐색  | PO의 가설 기반 시나리오               | 시나리오의 영향 범위를 수치화: "노쇼 발생 빈도가 월 87건"             |
| 5. 가정 검증    | PO에게 가정의 진위를 물어야 함           | **일부 가정을 데이터로 직접 검증**: 재진입 시점 분포로 "직후가 최적인가" 확인 |
| 6. 범위 합의    | "이 영역이 중요합니다"                | "이 영역의 이탈율이 X%이므로 Y명의 추가 전환이 기대됩니다"             |


### Phase 2에서의 영향도 판단 강화

ClickHouse 없이는 "경로상 위치"와 "현재 격차"만 코드로 판단 가능했습니다. ClickHouse가 연결되면 3번째 축이 추가됩니다:


| 판단 축               | 소스                     | 예시                                |
| ------------------ | ---------------------- | --------------------------------- |
| 경로상 위치 (필수/선택 경유지) | brownfield 코드 분석       | "홈 화면은 필수 경유지"                    |
| 현재 격차 (기능 존재 여부)   | brownfield 코드 분석       | "체험 완료 상태가 없음"                    |
| **실제 사용자 흐름**      | **ClickHouse 이벤트 데이터** | **"체험 완료→홈 재진입 72%, 홈→결제 도달 8%"** |


```
sprint-kit:
| 영역 | 경로 위치 | 현재 격차 | 실제 사용자 흐름 |
|------|----------|----------|----------------|
| 홈 화면 (체험 완료 후) | 필수 | 격차 큼 | 재진입 72% → 결제 도달 8% (이탈 64%p) |
| 결제 플로우 | 필수 | 격차 작음 | 도달 후 결제 완료 68% |
| 체험 결과 페이지 | 선택 | 격차 큼 | (페이지 자체 없음, 측정 불가) |
| 푸시 알림 | 선택 | 확인 필요 | 수신→재진입 23% |

데이터를 보면, 홈 화면에서의 64%p 이탈이 가장 큰 병목입니다.
이 분석에 동의하십니까?
```

### 쿼리 정책

`.sprint-kit.yaml`의 `query_policy`에 정의된 규칙을 따릅니다:

**허용되는 쿼리**:

- 집계 쿼리 (COUNT, SUM, AVG, 퍼널 전환율, 상태별 분포)
- 시점별 패턴 (이벤트 간 시간 분포)
- 예외 빈도 (노쇼, 취소, 소진 발생 건수)

**금지되는 쿼리**:

- 개별 사용자 행동 추적 (특정 user_id 기반 조회)
- PO가 요청하지 않은 범위의 탐색적 쿼리

### 데이터와 판단의 경계

데이터가 있어도 sprint-kit은 판정을 대신하지 않습니다. "탐지는 코드(+데이터), 판정은 PO" 원칙이 적용됩니다.

```
[올바른 사용]
sprint-kit: "체험 완료 후 재진입까지 평균 1.8일입니다. 분포: 당일 35%, 1일 후 28%, 2일 이후 37%.
            이 데이터를 보면, '완료 직후가 최적 시점'이라는 가정을 어떻게 판단하시겠습니까?"

[잘못된 사용]
sprint-kit: "데이터에 의하면 당일 재진입이 35%이므로, 완료 직후 유도가 최적입니다."
            → sprint-kit이 판정까지 수행 (금지)
```

### Graceful Degradation

ClickHouse가 연결되지 않으면, 데이터 활용 부분만 건너뛰고 나머지 Phase는 정상 진행됩니다. PO에게 "사용자 행동 데이터를 조회할 수 없으므로, 코드 분석과 PO님의 판단을 기반으로 진행합니다"라고 안내합니다.
