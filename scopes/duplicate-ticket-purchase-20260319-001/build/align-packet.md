## Align Packet: 수강권 중복 구매 허용 + 리텐션 프로모션 자동화

### 1. 당신이 요청한 것 (To-be)

**원문:** "현재 포도스피킹은 '1인 1언어 1수강권' 제약이 있어, 기존 회원이 동일 언어 수강권을 추가 구매할 수 없다. 이 변경은 중복 구매 제약을 제거하고, 만료 임박 고객에게 단계별 리텐션 프로모션을 자동 발송하여 이탈을 방지하는 것이 목적이다."

**시스템이 해석한 방향:**
레거시 연장수강권 시스템(팝업 결제 예약)을 제거하고, 중복 구매 시스템으로 교체한다. 기존 회원이 동일 언어 수강권을 미리 구매하여 보관할 수 있게 하고, 만료 임박 시점에 6단계 홈 팝업 + 4회 알림톡으로 재구매를 유도한다. 수강권 관리 UI를 개편하여 복수 수강권의 상태를 직관적으로 확인하고 관리할 수 있게 한다.

**제안된 범위:**

| 포함 | 동의하시나요? |
|------|-------------|

**시나리오:**

> S1: 기존 회원이 홈 "수강권 구매" 버튼 또는 팝업/알림톡을 통해 결제 페이지에 진입. ACTIVE 수강권이 있으면 구독 옵션 미노출, 일괄결제만 가능. 결제 완료 후 수강권은 INACTIVE(보관중)로 생성.
>
> S2: 만료 D-25부터 알림톡 4회 발송 — 리포트(D-25) → 쿠폰(D-14) → 리포트+긴급(D-7) → 쿠폰 긴급(H-12). 기존 심사 통과 템플릿 재사용. 중복 결제 완료 시 별도 알림 1종.
>
> S3: 만료 D-25부터 홈 팝업 6단계 — report(D-25~D-15, 학습통계 유도) → coupon-first(D-14, 쿠폰 첫 안내) → coupon-soft(D-13~D-10, 리마인드) → coupon-hard(D-7~D-2, 긴급 빨간색) → coupon-d1(D-1, 환불 보장) → coupon-d0(D-0, 최종 마감).
>
> S4: 수강권 관리 화면에서 활성화중/보관중/만료 3섹션으로 직관적 구분. 보관중 섹션에서 활성화 순서 드래그 앤 드롭. 각 수강권 카드 클릭 시 상세 내역(결제일, 기간 등). 하단 플로팅바로 언제든 수강권 구매 가능.
>
> S5: 환불은 CS 통해서만 처리. 개별 수강권 단위로 환불 산정. 앱 내 셀프 환불 없음.
>
> S6: 마이포도 메뉴에 "학습통계" 추가하여 상시 접근 가능. 기존 D-25 바텀시트 접근 경로도 유지.
>

### 2. 현재 현실 (As-is)

#### Experience 관점 — 지금 사용자가 보는 것

- **수강권 관리(마이포도)**: 사용중/홀딩중/종료예정/보너스/일괄결제/사용종료 6그룹 분류. "보관중(비활성)" 개념 없음.
- **홈 팝업**: 수강권 만료 D-25~D-0 구간에 연장 결제 유도 팝업 1종 표시. 시점별 차등 메시지 없이 동일 내용. 이것이 레거시 연장수강권 시스템의 진입점.
- **결제 페이지**: 신규 가입/체험 전환 고객 대상 설계. 기존 회원 전용 프로모션 배너/쿠폰 섹션 없음. 결제 완료 후 홈으로만 이동.
- **학습통계 리포트**: 존재하나 D-25 시점에만 홈 팝업 바텀시트로만 접근 가능. 상시 접근 경로 없음.

#### Policy 관점 — 지금 적용되는 규칙

- **약관 제15조 환불**: 개별 수강권 단위 환불 산정. 복수 보유 시 누적결제액 기준이 명시적으로 정의되지 않음.
- **약관 제16조 수강확인증**: 수강확인증 발급 수강권은 환불/해지 불가. 복수 보유 시 어느 수강권에 대한 발급인지 선택 절차 없음.
- **결제 정책**: 구독(BILLING) 결제는 즉시 ACTIVE 전환. 동일 언어 중복 구매 시 두 수강권이 동시 ACTIVE 되어 로직 충돌.
- **알림톡**: 기존 만료 리마인드 3종(D-25/D-7/당일). 정보성/광고성 발신키 분리. 기존 템플릿 심사 통과 상태.

#### Code 관점 — 지금 시스템이 할 수 있는 것

- **중복 구매 차단**: TicketServiceImpl에서 동일 언어 활성 수강권 2개 이상이면 InvalidTicketDuplicate 예외 throw. PaymentGateway에서 DUPLICATE_SUBSCRIPTION 검증으로 결제 자체를 차단.
- **수강권 상태**: 명시적 status enum 없음. 시작일/종료일/소진횟수/환불횟수 조합으로 활성 여부 판별. INACTIVE/DEPLETED/REFUNDED 상태 미정의.
- **알림톡**: 기존 pd_mkt_reg_remind 시리즈 템플릿으로 3종 발송. 신규 4종과 병행 시 중복 발송 위험 있으나, 기존 템플릿 재사용으로 위험 낮음.
- **홈 팝업**: PopupServiceImpl에서 D-0~D-25 범위 단일 팝업만 생성. phase별 분기 로직 없음.

#### CST-001 | Experience | 수강권 관리 UI에 비활성(INACTIVE) 상태 개념이 없음

**이것이 무엇인가:**
현재 마이포도 화면의 수강권 목록은 사용중/홀딩중/종료예정/보너스/일괄결제/사용종료 6그룹으로 분류됩니다. "보관중(비활성)" 상태가 존재하지 않습니다.

**왜 충돌하는가:**
중복 구매 후 INACTIVE 상태로 보관되는 수강권이 생기면, 현재 UI에서는 이를 표시할 그룹이 없습니다.

**처리하지 않으면:**
비활성 수강권이 UI에 표시되지 않아 고객이 보관 중인 수강권을 확인/활성화할 수 없음

**변경 규모:** 화면 구조 재설계 — 기존 6그룹을 3섹션(활성화중/보관중/만료)으로 개편

---

#### CST-002 | Experience | 홈 팝업이 단일 구조 — 6단계 phase별 분기 없음

**이것이 무엇인가:**
PopupServiceImpl에서 D-0~D-25 범위에 동일한 연장 결제 유도 팝업 1종만 생성합니다.

**왜 충돌하는가:**
6단계 phase별 차등 메시지(리포트→쿠폰→긴급)를 전달하려면 phase 분기 로직이 필요합니다.

**처리하지 않으면:**
만료 임박 시점별 차등 메시지(리포트→쿠폰→긴급)를 전달할 수 없어 리텐션 효과 감소

**변경 규모:** 백엔드 팝업 서비스 + 프론트 바텀시트 6종 구현

---

#### CST-003 | Experience | 결제 페이지가 기존 회원 프로모션 미지원

**이것이 무엇인가:**
현재 결제 페이지는 신규 가입/체험 전환 고객 대상으로 설계되어 있습니다.

**왜 충돌하는가:**
기존 회원이 프로모션(쿠폰, 할인) 혜택을 적용하여 결제하려면 별도의 프로모션 배너/쿠폰 섹션이 필요합니다.

**처리하지 않으면:**
기존 회원이 프로모션 혜택을 인지하지 못해 전환율 저하

**변경 규모:** 결제 페이지에 기존 회원 전용 경로(/payment/existing-user) 추가

#### CST-004 | Code | InvalidTicketDuplicate 예외 — getActiveTicket() 전면 수정 필요

**이것이 무엇인가:**
TicketServiceImpl에서 동일 언어 활성 수강권이 2개 이상이면 InvalidTicketDuplicate 예외를 throw합니다. getActiveTicket()은 단일 Ticket 반환을 전제로 합니다.

**왜 충돌하는가:**
중복 구매를 허용하면 동일 언어에 복수 수강권이 존재하게 되어, getActiveTicket()에 의존하는 예약/수업/환불 로직이 모두 영향을 받습니다.

**처리하지 않으면:**
중복 구매 허용 시 기존 예약/수업/환불 등 getActiveTicket()에 의존하는 모든 로직이 오류 발생

**변경 규모:** 핵심 서비스 메서드 수정 + 의존 로직 전수 점검

#### CST-006 | Code | TicketStatus enum 미정의 — 5상태 전이 기반 없음

**이것이 무엇인가:**
Ticket 엔티티에 명시적 status 필드가 없습니다. 날짜/카운트 조합으로 상태를 판별합니다.

**왜 충돌하는가:**
5상태 전이(INACTIVE/ACTIVE/EXPIRED/DEPLETED/REFUNDED)를 구현하려면 명시적 상태 관리 기반이 필요합니다.

**처리하지 않으면:**
5상태 전이 로직을 구현할 기반이 없어 비활성 보관, 횟수 소진, 환불 상태를 구분할 수 없음

**변경 규모:** Ticket 도메인 모델에 status enum 도입 + 기존 판별 로직 마이그레이션

#### CST-007 | Code | 기존 알림톡 3종과 신규 4종 병행 시 중복 발송 위험

**이것이 무엇인가:**
기존 pd_mkt_reg_remind 시리즈 3종과 신규 pd_lrn_* 시리즈 4종이 동시 활성화되면 같은 고객에게 중복 발송됩니다.

**왜 충돌하는가:**
기존 템플릿을 재사용하므로 위험이 낮으나, 전환 시점에 기존 스케줄러 비활성화가 필요합니다.

**처리하지 않으면:**
같은 고객에게 동일 시점에 구버전+신버전 알림톡이 중복 발송되어 고객 피로도 증가

**변경 규모:** 배포 시점에 기존 템플릿 비활성화 + 신규 스케줄러 활성화

#### CST-008 | Policy | 약관 제15조 환불 — 복수 수강권 산정 기준 모호

**이것이 무엇인가:**
약관 제15조의 환불 공식이 단일 수강권을 전제로 합니다. 복수 보유 시 originPrice/누적결제액 기준이 정의되지 않았습니다.

**왜 충돌하는가:**
복수 수강권 환불 시 어떤 수강권의 결제액을 기준으로 산정하는지 모호합니다.

**처리하지 않으면:**
복수 수강권 환불 시 누적결제액 산정이 모호해 CS 분쟁 발생 가능

**변경 규모:** CS 운영 가이드에 개별 수강권 단위 환불 기준 명시 (앱 변경 없음)

#### CST-009 | Policy | 수강확인증 발급 시 수강권 선택 필요

**이것이 무엇인가:**
약관 제16조 4항에 따라 수강확인증 발급 수강권은 환불/해지가 불가합니다. 복수 보유 시 어느 수강권에 대한 발급인지 선택해야 합니다.

**왜 충돌하는가:**
현재 수강확인증 발급 UI에 수강권 선택 기능이 없습니다.

**처리하지 않으면:**
수강확인증 발급 대상이 불명확하면 의도치 않은 수강권의 환불이 차단될 수 있음

**변경 규모:** 수강확인증 발급 UI에 수강권 선택 드롭다운 추가

---

#### CST-010 | Policy | 구독(BILLING) 즉시 ACTIVE — 일괄결제만 허용 필요

**이것이 무엇인가:**
구독(BILLING) 결제는 결제 즉시 수강권이 ACTIVE로 전환됩니다. 동일 언어에 ACTIVE가 2개 되면 로직 충돌이 발생합니다.

**왜 충돌하는가:**
중복 구매 수강권을 INACTIVE(보관)로 유지하려면 일괄결제(LUMP_SUM)만 허용해야 합니다.

**처리하지 않으면:**
구독 결제로 중복 구매 시 두 수강권이 동시 ACTIVE 되어 예약/수업/환불 로직 충돌

**변경 규모:** 결제 페이지에서 ACTIVE 수강권 존재 시 구독 옵션 미노출

---

#### CST-011 | Policy | 카카오 알림톡 정보성/광고성 분류 미확정

**이것이 무엇인가:**
리텐션 알림톡이 카카오 심사에서 광고성으로 분류되면 마케팅 수신 동의 유저만 대상이 됩니다.

**왜 충돌하는가:**
기존 심사 통과 템플릿을 재사용하므로 실질적 충돌 없음 (PO 확인).

**처리하지 않으면:**
카카오 심사에서 광고성으로 판정되면 발송 대상이 대폭 축소되어 리텐션 효과 감소

**변경 규모:** 기존 템플릿 재사용으로 추가 작업 없음
