## Align Packet: 무료체험 횟수 정책 변경 (1회 → 3회)

### 1. 당신이 요청한 것 (To-be)

**원문:** "무료체험 횟수 정책 변경 (1회 → 3회). 1회 제공 시 노쇼·튜터 노쇼 등으로 체험 기회가 소멸되어 결제 전환 모수가 줄어드는 문제를 해결한다. 3회 제공으로 서비스 가치를 충분히 체험할 기회를 확보하고, 결제 전환 유도 대상을 넓힌다. 회차별 개방 조건(직전 회차 정상 종료 시 다음 회차 개방), 학생 노쇼·취소 시 차감 규칙, 튜터 노쇼 시 복구 규칙, 3회 소진 후 유료 체험(5,000원/3회) 전환 포함."

**시스템이 해석한 방향:**
체험 레슨을 1회에서 3회로 확대하고, 회차별 개방·차감·복구 규칙을 정의하며, 3회 소진 후 유료 체험(5,000원/3회)으로 전환하는 정책 변경을 백엔드·프론트엔드·정책 문서에 걸쳐 반영한다.

**시나리오:**

> 시나리오 1: 신규 학생이 가입 후 무료 체험 3회를 신청한다. 첫 번째 체험 수업을 예약하고, 정상 완료 후 두 번째 체험이 개방된다. 세 번째까지 완료하면 AFTER_TRIAL 쿠폰이 발급되고 정규 결제 유도 화면이 표시된다.
>
> 시나리오 2: 학생이 첫 번째 체험을 노쇼한다. 해당 회차가 차감(또는 복구)되고, 남은 체험 횟수가 안내된다. 차감된 경우 2회만 남은 상태로 다음 체험을 예약할 수 있다.
>
> 시나리오 3: 튜터가 노쇼한다. 해당 회차가 복구되어 학생에게 재신청 화면이 표시되고, 잔여 횟수는 그대로 유지된다.
>
> 시나리오 4: 학생이 첫 번째 체험만 완료한 상태에서 정규 구독을 결제한다. 미완료 체험 2회가 자동 취소되고 정규 수강권이 활성화된다.
>
> 시나리오 5: 학생이 무료 체험 3회를 모두 소진한 후, 유료 체험(5,000원/3회)을 구매하여 추가 체험을 진행한다.
>

### 2. 현재 현실 (As-is)

#### Experience 관점 — 지금 사용자가 보는 것

현재 체험 레슨은 1회 제공입니다. 신규 가입 시 홈 화면에 체험 배너가 노출되고, 체험 결제(0원 또는 500원) → 수업 예약 → 수업 완료 → 결제 유도 페이지로 강제 리다이렉트되는 선형 플로우입니다. 체험 완료/미완료는 trialClassCompYn(Y/N) 이진 플래그로 관리되며, '진행 중' 상태가 없습니다. 취소 다이얼로그에는 '다시 체험 레슨을 들을 수 없어요'라는 1회 전제 경고가 표시됩니다.

#### Policy 관점 — 지금 적용되는 규칙

구독 정책에 '1인 1회 체험 제한'이 명시되어 있고, DuplicateLessonValidator가 기존 체험 수업 존재 시 추가 구매를 차단합니다. 체험 수강권 만료일은 7일(하드코딩), 이용약관은 매일 1회 예약(시작 2시간 전)으로 제한합니다. 노쇼 시 수강권이 무조건 차감(nUsed += 1)되며, 체험권 전용 예외 규칙은 없습니다. 체험 완료 후 AFTER_TRIAL 쿠폰이 자동 발급되고, 정규 결제 시 체험 수업이 일괄 취소(L7/L8)됩니다.

#### Code 관점 — 지금 시스템이 할 수 있는 것

체험 수강권 발급 시 nPurchased=1이 하드코딩되어 있고, 체험 결제 시 수업(GT_CLASS)이 1건만 자동 생성됩니다. Student 엔티티의 trialPaymentYn(체험 결제 여부)은 Y/N 이진값으로, '3회 중 일부 사용' 상태를 표현할 수 없습니다. 프론트엔드의 배너 노출, 리다이렉트, Greeting 위젯 등이 이 이진 플래그에 의존합니다. 체험 완료 판정(hasCompletedTrialClass)도 1건이라도 DONE이면 true를 반환합니다.

#### CST-001 | Policy | 1인 1회 체험 제한 정책이 구독 정책·검증 로직·DB에 걸쳐 적용 중이며, 3회 허용으로 전면 변경 필요

**이것이 무엇인가:**
구독 정책·검증 로직·DB 전반에 '체험 = 1회' 제한이 적용되어 있음

**왜 충돌하는가:**
3회 체험을 허용하려면 정책 문서(subscription/policies.md), 검증 로직(DuplicateLessonValidator), Student.trialPaymentYn 필드 구조를 모두 변경해야 함

**처리하지 않으면:**
결제 검증(DUPLICATE_SUBSCRIPTION)에서 2회차 이후 체험 신청이 차단됨. trialPaymentYn=Y 전환 후 추가 체험 불가

**변경 규모:** 백엔드 정책 + DB 스키마 + 검증 로직 전반 수정. 영향 범위 넓음

#### CST-003 | Policy | 체험 수강권 만료일이 7일(하드코딩)인데, 매일 1회 예약 제한(이용약관) 감안 시 3회 소화에 최소 3일 필요 — 만료일 연장 여부 결정 필요

**이것이 무엇인가:**
체험 수강권 만료일이 7일로 하드코딩됨

**왜 충돌하는가:**
매일 1회 예약 제한 + 최소 2시간 전 예약 규칙을 적용하면 3회 소화에 최소 3~4일 필요. 노쇼 시 차감되면 재예약에 추가 일수 필요

**처리하지 않으면:**
학생이 7일 내 3회 예약을 완료하지 못하면 체험권이 만료되어 잔여 횟수를 사용할 수 없음

**변경 규모:** TrialPaymentProcessor.java의 giveDays 값 1곳 변경 + 프론트엔드 유효기간 표시 변경

**선택지:**

| 선택 | 이점 | 리스크 | 내용 |
|------|------|--------|------|
| 7일 유지 | 기존 정책 변경 없음 | 여유 부족. 노쇼/일정 변경 시 3회 미소화 가능 | 현재 이용약관 '매일 1회' 제한 감안 시 최소 3일 필요. 7일이면 충분해 보이나 노쇼 시 여유 감소 |
| 14일로 연장 | 3회 소화에 충분한 시간 확보 | 체험 기간 연장으로 정규 전환 유도 지연 |  |
| 21일로 연장 | 최대 여유 확보 | 체험 기간이 너무 길어 전환율 하락 우려 |  |

**추천:** 7일 유지 권장 — 이론상 3~4일이면 3회 소화 가능. 단, 노쇼 복구 정책(CST-005)과 함께 검토 필요

#### CST-004 | Policy | 무료 3회 체험의 결제 유형을 TRIAL(카드 파킹 포함)과 TRIAL_FREE(0원 결제) 중 결정 필요 — 유료 체험(5,000원/3회) 전환 시 결제 유형도 함께 결정

**이것이 무엇인가:**
체험 결제 유형이 TRIAL(유료, 카드 파킹)과 TRIAL_FREE(무료, 0원 결제)로 이원화

**왜 충돌하는가:**

**처리하지 않으면:**
결제 유형에 따라 카드 정보 저장 여부, 체험 종료 스케줄 등록 여부가 달라지므로 올바른 유형을 선택하지 않으면 결제 전환 파이프라인이 정상 동작하지 않음

**변경 규모:** 결제 프로세서 분기 + 프론트엔드 결제 UI 변경

**선택지:**

| 선택 | 이점 | 리스크 | 내용 |
|------|------|--------|------|
| TRIAL_FREE 사용 (무료 3회 전체) | 결제 부담 없음. 진입 장벽 최소화 | 카드 정보 미수집. 정규 전환 시 결제 정보 재입력 필요 |  |

**추천:** brief 의도(전환 모수 확대)를 고려하면 TRIAL_FREE가 적합. 단, 유료 체험(5,000원) 전환 시에는 TRIAL 유형 사용
