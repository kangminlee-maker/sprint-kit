## Align Packet: 홈 화면 체험 상태 기반 동적 전환

### 1. 당신이 요청한 것 (To-be)

**원문:** "홈 화면을 학생의 체험 진행 상태(미신청·대기·레슨 중·완료·예외)에 따라 동적으로 전환되는 구조로 재설계. 각 단계에서 다음 행동(신청 → AI학습 체험 → 레슨 입장 → 결제)을 명확히 유도하고, 예외 상황에서도 결제 유도까지의 경로 보장."

**시스템이 해석한 방향:**
홈 화면의 상태 분기 로직을 현재 4가지(수강권 없음/예약됨/체험 예약 유도/정규 예약 유도)에서 체험 흐름 전체(미신청→대기→레슨 중→완료→예외)를 커버하는 구조로 확장하고, 각 상태에 맞는 메인 문구·프로그레스바·본문 카드·플로팅 CTA를 구성한다.

**제안된 범위:**

| 포함 | 동의하시나요? |
|------|-------------|
| 홈 화면 상태 분기 로직 확장 (4가지 → 6~7가지 세분화) | |
| 체험 프로그레스바 컴포넌트 신규 구현 (신청→예습→레슨→완료 단계 표시) | |
| 상태별 본문 카드 영역 콘텐츠 전환 (인사 카드, 예습 진행률, 레슨 입장, 결제 혜택 카드) | |
| 하단 플로팅 CTA 버튼 상태별 라벨·동작 전환 | |
| 예외 상태 처리 (튜터 노쇼 → 재신청, 학생 노쇼/취소 → 잔여 횟수 안내, 3회 소진 → 유료 체험 안내) | |
| GNB "수강신청" 버튼을 통한 상시 결제 접근 경로 유지 | |

| 제외 | 동의하시나요? |
|------|-------------|
| 백엔드 API 신규 개발 (기존 API의 응답 데이터 활용 범위 내에서 해결, 부족 시 별도 scope) | |
| 정규 수강생(결제 완료 후) 홈 화면 변경 | |
| 레슨룸(강의실) 내부 UI 변경 | |
| 예약/결제 페이지 자체의 UI 변경 | |
| AI 학습(스마트톡) 페이지 변경 | |
| 노쇼 판정 로직 변경 (외부 시스템 영역) | |

**시나리오:**

> 시나리오 1: 신규 학생이 앱에 로그인하면 홈 화면에 "무료 체험 레슨 신청하기" 프로그레스바 첫 단계가 활성화되고, 플로팅 버튼으로 신청 플로우에 진입한다.
>
> 시나리오 2: 체험 레슨을 예약한 학생이 홈 화면에 진입하면 프로그레스바가 "대기" 단계로 전환되고, 예습 진행률과 레슨 일정이 표시되며, 시작 10분 전에 플로팅 버튼이 "레슨 입장"으로 변경된다.
>
> 시나리오 3: 체험 레슨을 정상 완료한 학생의 홈 화면에는 프로그레스바가 "완료" 단계를 표시하고, 첫 수강 혜택 카드와 프로모션 배너가 노출되며, 플로팅 버튼이 "수강권 구매"로 전환된다.
>
> 시나리오 4: 튜터 노쇼로 체험이 비정상 종료된 학생에게는 체험 1회 복구 안내와 함께 "다시 신청하기" 버튼이 표시된다. 무료체험에는 72시간 예약 차단 패널티가 적용되지 않으므로 즉시 재신청이 가능하다.
>
> 시나리오 5: 무료체험 3회를 모두 소진한 학생에게는 유료 체험(5,000원/3회) 안내 카드와 결제 CTA가 표시된다.
>

### 2. 현재 현실 (As-is)

#### Experience 관점 — 지금 사용자가 보는 것

현재 홈 화면은 4가지 상태로 분기한다: (1) 수강권 없음(NoTicketState) — "수강권 둘러보기" 버튼만 표시, (2) 체험 예약 유도(RecommendTrialLessonState) — "예약하기" 버튼, (3) 정규 예약 유도(RecommendRegularLessonState) — "예약하기" 버튼, (4) 예약된 레슨(ScheduledClassState) — 카운트다운, 예습 진행률, 입장/변경 버튼. 체험 완료 후 결제 유도, 노쇼/취소 후 재신청, 3회 소진 후 유료 전환에 해당하는 화면 상태가 없다. 프로그레스바도 없다.

#### Policy 관점 — 지금 적용되는 규칙

이용약관 제14조에 따라 노쇼 패널티(무제한권 72시간 예약 차단, 회차권 회차 미복구)가 존재하지만, 무료체험은 수강권과 별개 상품이므로 이 패널티가 적용되지 않는다. 튜터 노쇼 시에는 체험 1회가 복구된다. 체험수업은 일시정지(홀딩) 불가하며, 정규구독 결제 시 체험수업이 자동 만료된다. 무료체험은 TRIAL_FREE, 유료체험은 TRIAL로 PaymentType이 구분된다.

#### Code 관점 — 지금 시스템이 할 수 있는 것

홈 화면 상태 판별은 useGreetingStatus 훅이 담당하며, getSubscribeMappList(수강권 목록)와 getNextLessonsInfo(다음 레슨 정보) 두 API 응답만 사용한다. 레슨의 세부 상태(완료/노쇼/취소)는 invoiceStatus 필드에 기록되나, 현재 홈 화면에서 이 필드를 조회하지 않는다. 노쇼 처리는 레거시 PHP/어드민 도구가 담당하며 Java 백엔드에는 해당 전이 로직이 없다.

<details>
<summary>기술 상세 (Builder 참고용)</summary>

- Lesson 엔티티: invoiceStatus(InvoiceStatus enum 8값), status(LectureStatus 4값), classState(String 6값)의 3중 상태 구조
- InvoiceStatus 전이: CREATED → RESERVED → COMPLETED/NOSHOW_S/NOSHOW_BOTH/CANCEL_NOSHOW_T/CANCEL/CANCEL_PAID
- SubscribeMappStatus: TRIAL/SUBSCRIBE/LUMP_SUM/EXTEND/BONUS/FAILED/FINISH/REFUND/ADMIN_DROP/DELETE

- CurriculumType: TRIAL, TRIAL_1, TRIAL_2 (체험 변형)
- podo-app: Next.js 15 + React 19, FSD 아키텍처(features/widgets/entities/shared), tanstack-query로 서버 상태 관리
- 디자인 시스템: vibe-design(Tailwind v4 기반), podo-app은 design-system-temp 패키지 사용
- GNB 탭: 홈, 레슨, 예약, AI 학습, 마이포도 (현재 "수강신청" 탭은 없고 "레슨" 탭이 /subscribes로 연결)
</details>

### 3. 충돌 지점 (Tension)

요청한 것(to-be)과 현재 현실(as-is) 사이에 4건의 충돌이 발견되었습니다.

| CST-ID | 관점 | 요약 |
|--------|------|------|
| CST-001 | Experience | 현재 홈 화면 상태 분기가 4가지뿐이어서 brief가 요구하는 세분화된 체험 상태(6~7가지)를 표현할 수 없음 |
| CST-004 | Code | 체험 레슨 상태 판별에 필요한 API 응답 데이터가 현재 홈 화면 쿼리에 부재 — 노쇼 여부, 잔여 체험 횟수, 체험 완료 후 결제 상태를 별도 조회해야 함 |
| CST-007 | Code | SubscribeMappStatus에 TRIAL 상태가 있으나 무료체험 횟수(3회) 추적과 유료체험(5,000원/3회) 전환을 위한 전용 필드/API가 코드에서 확인되지 않음 |
| CST-009 | Policy | 체험수업은 정규구독 결제 시 자동 만료(취소)되는 정책이 존재 — 홈 화면에서 체험 상태와 결제 유도를 동시에 표시할 때 상태 충돌 가능 |

---

#### CST-001 | Experience | 현재 홈 화면 상태 분기가 4가지뿐이어서 brief가 요구하는 세분화된 체험 상태(6~7가지)를 표현할 수 없음

**이것이 무엇인가:**
현재 홈 화면 상태 분기(greetingStatusSchema)가 4가지로 고정되어 있다. brief가 요구하는 체험 세부 상태(미신청/대기/레슨 중/완료/노쇼/취소/소진)를 표현하려면 상태 모델을 근본적으로 재설계해야 한다.

**왜 충돌하는가:**
기존 useGreetingStatus는 "수강권 유무"와 "다음 레슨 유무"라는 2가지 축으로만 분기한다. brief가 요구하는 체험 흐름은 레슨의 종료 상태(완료/노쇼/취소)와 잔여 체험 횟수까지 포함하므로 분기 축이 4~5개로 증가한다.

**처리하지 않으면:**
기존 4가지 상태(NO_TICKET, SCHEDULED_CLASS, RECOMMEND_BOOKING_TRIAL, RECOMMEND_BOOKING_REGULAR)로는 체험 완료 후 결제 유도, 예외(노쇼/취소/소진) 상태를 표현할 수 없어 홈 화면 재설계의 핵심 목표를 달성할 수 없음

**변경 규모:** 프론트엔드 홈 화면 핵심 로직 재작성. 상태 모델, 훅, 컴포넌트 전면 수정.

**선택지:**

| 선택 | 이점 | 리스크 | 내용 |
|------|------|--------|------|
| 기존 greetingStatusSchema 확장 | 기존 구조 유지, 점진적 변경 가능 | enum 값이 7개 이상으로 증가하며 조합 복잡도 상승 | greetingStatusSchema에 TRIAL_COMPLETED, TRIAL_NOSHOW, TRIAL_EXHAUSTED 등 추가 |
| 계층적 상태 모델 도입 | 체험/정규를 분리하여 각각 독립적 세부 상태 관리 | 기존 코드와의 호환성 단절, 전면 재작성 필요 | 상위 분기(체험/정규/미가입) → 하위 분기(세부 상태)의 2계층 구조 |

---

#### CST-004 | Code | 체험 레슨 상태 판별에 필요한 API 응답 데이터가 현재 홈 화면 쿼리에 부재 — 노쇼 여부, 잔여 체험 횟수, 체험 완료 후 결제 상태를 별도 조회해야 함

**이것이 무엇인가:**
홈 화면에서 호출하는 2개 API(getSubscribeMappList, getNextLessonsInfo)의 응답에 체험 레슨의 종료 상태(완료/노쇼/취소)와 잔여 무료체험 횟수 정보가 포함되지 않는다.

**왜 충돌하는가:**
brief의 모든 상태 전환(완료 후 결제 유도, 노쇼 후 재신청, 3회 소진 후 유료 전환)은 이 데이터에 의존한다. 데이터 없이는 상태 분기 자체가 불가능하다.

**처리하지 않으면:**
현재 getSubscribeMappList와 getNextLessonsInfo 두 API만으로는 체험 완료/노쇼/취소/소진 상태를 구분할 수 없음. 백엔드 API 확장 또는 추가 엔드포인트 호출 필요

**변경 규모:** 프론트엔드 API 호출 추가 또는 백엔드 API 응답 확장. 범위에 따라 백엔드 scope 분리 필요.

**선택지:**

| 선택 | 이점 | 리스크 | 내용 |
|------|------|--------|------|
| 기존 API 응답 확장 요청 | 프론트엔드 호출 횟수 유지, 네트워크 비용 최소 | 백엔드 수정이 이 scope 범위를 넘을 수 있음 | getSubscribeMappList 응답에 체험 레슨 상태·잔여 횟수 필드 추가 |
| 프론트엔드에서 추가 API 호출 | 백엔드 수정 없이 기존 API 조합으로 해결 가능성 | API 호출 횟수 증가, 데이터 정합성 관리 복잡 | invoiceStatus를 반환하는 레슨 목록 API를 추가 호출하여 프론트엔드에서 조합 |

#### CST-007 | Code | SubscribeMappStatus에 TRIAL 상태가 있으나 무료체험 횟수(3회) 추적과 유료체험(5,000원/3회) 전환을 위한 전용 필드/API가 코드에서 확인되지 않음

**이것이 무엇인가:**
무료체험 횟수(3회 한도) 추적과 유료체험(5,000원/3회) 전환을 위한 데이터 구조가 코드에서 명확히 확인되지 않는다.

**왜 충돌하는가:**
brief는 "3회 전체 소진 시 유료 체험을 통해 결제 유도 화면에 반드시 도달"을 요구한다. 이를 구현하려면 현재 학생의 무료체험 사용 횟수, 잔여 횟수, 유료체험 전환 여부를 정확히 파악해야 한다.

**처리하지 않으면:**
"무료체험 3회 소진 → 유료 체험 → 결제 유도"의 전환 로직을 구현하려면 잔여 무료체험 횟수를 추적하는 데이터 구조가 필요. 현재 코드에서 이 구조가 명확하지 않아 백엔드 수정 범위 확인 필요

**변경 규모:** 데이터 구조 확인 및 필요 시 백엔드 확장. 확인 결과에 따라 범위 변동.

**선택지:**

| 선택 | 이점 | 리스크 | 내용 |
|------|------|--------|------|
| PaymentType(TRIAL/TRIAL_FREE)와 수강권 이력으로 추적 | 기존 데이터로 해결 가능성 | 이력 조회 로직이 복잡해지고, 정확한 횟수 계산 보장 어려움 |  |
| 백엔드에 체험 횟수 전용 필드/API 추가 | 정확한 횟수 관리, 프론트엔드 로직 단순화 | 백엔드 수정 범위 확대 | 사용자별 무료체험 사용/잔여 횟수를 반환하는 전용 엔드포인트 |

#### CST-009 | Policy | 체험수업은 정규구독 결제 시 자동 만료(취소)되는 정책이 존재 — 홈 화면에서 체험 상태와 결제 유도를 동시에 표시할 때 상태 충돌 가능

**이것이 무엇인가:**
정규구독 결제 시 체험수업이 자동 만료(취소)된다.

**왜 충돌하는가:**
brief는 체험 완료 후 결제 CTA를 전면 노출한다. 학생이 체험 진행 중 GNB의 수강신청 버튼을 통해 결제를 완료하면, 체험 수업이 자동 취소되어 홈 화면 상태가 즉시 전환된다. 이 전환이 매끄럽지 않으면 사용자가 혼란을 느낀다.

**처리하지 않으면:**
학생이 체험 진행 중 결제하면 체험 수업이 자동 취소됨(L7, S5 전이). 홈 화면에서 체험 진행 상태와 결제 CTA를 동시에 보여줄 때, 결제 완료 후 체험 상태가 즉시 사라지는 UX 전환을 고려해야 함
